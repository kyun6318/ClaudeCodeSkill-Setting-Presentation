# Skills Demo Plan — "Weekly Note Hub"

> 발표 `PRESENTATION-PLAN.md` / `PRESENTATION-OUTLINE.md` / `SKILL-NAME-MAPPING.md`의 스킬 묶음을 하나의 예제 프로젝트에 **두루두루** 얹어, 각 스킬의 장점이 동작으로 드러나게 한다.
> 시연 영상·슬라이드 스니펫의 원재료(raw material) 역할. 실제 발표에서는 5장(사전 녹화 영상) / 7장(설정 스니펫) / 8장(효과·한계 수치 참조)에 소비된다.

## 1. 예제 주제: "Weekly Note Hub"

개발자 개인이 주간 학습·회고 노트를 저장하고 태그·검색으로 다시 꺼내 보는 작은 대시보드.

### 왜 이 주제인가

| 기준 | 근거 |
|---|---|
| **도메인 중립** | 청중 직군(BE/FE/시스템)·소속 도메인과 무관하게 공감 가능 |
| **적절한 규모** | 3~4개 화면, CRUD + 검색 + 간단 통계. 한 사람이 1~2일 내 빌드 가능 |
| **단계별 결함이 실제로 나타남** | "대충 말하면" 기획·명세·검증·보안 모두에서 실패가 재현됨 → 5가지 결함(6장) 시연 소재 |
| **자율 개선 여지** | 검색 품질/요약 정확도 지표가 있어 `/self-improvement-loop` 대상이 됨 |
| **메타-레퍼런스 회피** | "Claude 사용 로그 분석" 같은 메타 주제는 흥미롭지만 청중이 따라 만들기 어려움. 이 주제는 누구나 이식 가능 |

### 스택 (PRESENTATION-CONTEXT 비공개 원칙과 정합)

- Next.js 14 (App Router) + TypeScript
- Supabase (auth + DB) — 기존 `project-bootstrap-supabase` 스킬 활용 포인트 생성
- Tailwind + shadcn/ui
- Vitest + Playwright (검증용)

### 화면 구성 (최소)

1. 노트 목록 (검색/태그 필터)
2. 노트 작성·편집
3. 노트 상세 + AI 요약 패널
4. 주간 대시보드 (태그 분포·작성 빈도)

## 2. 발표 메시지와의 연결

예제는 "스킬 자랑"이 아니라 **발표 Key Takeaway 3줄을 동작으로 증명**하기 위한 도구다.

| Key Takeaway | 예제에서 드러내는 방식 |
|---|---|
| ① 에이전트는 사람을 대체하지 않는다 | 1차 "만들어줘" 요청 → 실패 재현(2장 사례). 같은 작업을 스킬 파이프라인으로 다시 돌리면 성공 |
| ② 잘 쓰려면 운영 원칙이 필요하다 | 기획·명세·검증·핸드오프 단계를 건너뛸 때 어떤 결함이 나오는지를 **의도적으로 한 번 보여준 뒤** 스킬로 막는다 |
| ③ 스킬은 원칙을 반복 가능하게 고정한다 | 동일 요구로 두 번 실행 → 일관된 결과. `/self-improvement-loop`로 keep/revert 결정 장면 |

## 3. 전체 파이프라인 (스킬 소비 순서)

발표 6장 파이프라인 다이어그램 그대로, 단계별로 **그 단계에서 드러낼 장점 한 줄**을 고정한다.

```
아이디어 → 기획 → 명세 → 태스크 → 실행 → 검증 → 자율 개선
   ↑       ↑      ↑       ↑       ↑       ↑         ↑
  ①       ②      ③       ④       ⑤       ⑥         ⑦
```

| # | 단계 | 사용 스킬 (직관 이름) | 드러낼 장점 (한 줄) |
|---|---|---|---|
| ① | 아이디어 수렴 | `/idea-concretizer`, `/co-brainstorming` | 추상적 한 줄("주간 노트 허브")을 구체 MVP 3~4개 제안으로 벌려준다 |
| ② | 요구사항 수렴 | `/requirements-dialogue` | 대충 말하면 AI가 알아서 결정하던 것들을 **질문으로 되돌려 사용자 결정**하게 한다 |
| ②' | 숨은 가정 해체 | `/hidden-assumption-surfacer`, `/technical-premise-deconstructor` | "Supabase 쓸 거죠?", "실시간 동기화가 필요하다 가정했죠?"를 **명시화** |
| ③ | 화면 명세 | `/screen-spec-generator` | 화면 단위 YAML 명세가 있어야 뒤 단계가 추측으로 가지 않는다 |
| ④ | 태스크 분해 | `/task-decomposer` | Domain-Guarded 분해 — 화면 요구와 도메인 리소스의 Interface Contract 검증 |
| ⑤-a | 자동 실행 | `/dependency-auto-executor` | 의존성 기반 직렬/병렬 — 사람이 체크박스 돌보지 않아도 됨 |
| ⑤-b | (대체 경로) 풀 파이프라인 | `/full-pipeline-orchestrator` | 아이디어→빌드까지 한 번에. "한 방에 가고 싶을 때"의 체감 |
| ⑤-c | (대체 경로) Build-Eval 하네스 | `/autonomous-build-autonomous-build-harness` | spec만 있으면 Evaluator가 직접 테스트하며 수정 루프 |
| ⑤-d | 초슬림 모드 | `/ultra-thin-executor` | 태스크가 많을 때 오토 컴팩팅 없이 진행 |
| ⑥-a | 완료 전 검증 | `/pre-completion-verifier` | "완성됨"을 **증거로 대체** — 빌드·타입·테스트·런타임 로그 |
| ⑥-b | 품질 게이트 | `/quality-gate` | 메트릭·리포트로 Phase 완료 판정 자동화 |
| ⑥-c | 2단계 코드 리뷰 | `/code-reviewerer` | 스펙 준수 → 코드 품질의 순서. 스펙이 빠지면 품질 판단이 헛돈다 |
| ⑥-d | 버그 재현 | `/systematic-debugger` | 4단계 근본원인 분석 — "수정 전에 원인부터" |
| ⑥-e | 보안 감사 | `/auto-security-audit` | STRIDE + OWASP Top10 자율 탐색. 인증 붙는 시점에 의무화 |
| ⑥-f | 자동 에러 수정 | `/auto-error-fixer` | Build > Type > Test > Lint 순서. 에러 폭탄이 떨어졌을 때의 1차 대응 |
| ⑦ | 자율 개선 | `/self-improvement-loop` | Frozen Metric + keep/revert — "일회성 생성"을 막는다 |
| — | 운영 메타 | `/token-compressor`, `/cost-model-router` | 위 전 과정을 깔고 들어가는 **공통 축소층** — 같은 결과를 더 싸게 |

> **시연 무게**: ②③⑤-a⑥-a⑦ 다섯 개가 5장 영상의 핵심 뼈대. 나머지는 스크린샷·단문 언급으로 장점만 집어준다.

## 4. 장점이 "대비로" 드러나는 장면 설계

스킬별 장점은 **Before(스킬 없이) vs After(스킬 사용)** 대비로 가장 잘 드러난다. 각 스킬마다 1개 장면을 사전에 녹화해둔다.

### 4.1 /requirements-dialogue — 수렴 vs 확산

| Before | After |
|---|---|
| "주간 노트 허브 만들어줘" → AI가 Firebase로 시작, 사용자는 Supabase 기대 | `/requirements-dialogue`로 10~15개 질문 → 스택/인증/저장 정책 사용자 결정 |
| **결함**: 숨은 가정 누적 | **장점**: 결정권이 사람에게 남음 |

### 4.2 /hidden-assumption-surfacer — 가정의 명시화

- 같은 스펙을 AI에게 두 번 넣었을 때 서로 다른 결과가 나오는 장면을 스크린샷(2장 재사용 가능)
- 중간에 `/hidden-assumption-surfacer`를 끼우면 "AI가 암묵적으로 깔고 있던 가정" 목록이 먼저 튀어나옴
- **장점**: 드리프트 사전 차단

### 4.3 /screen-spec-generator — 명세가 있어야 실행이 선다

- 명세 없이 구현 → 4번째 화면에서 데이터 스키마가 달라져 재작업
- 명세 있음 → 태스크 분해가 같은 리소스를 참조

### 4.4 /dependency-auto-executor — 의존성 기반 자동 실행

- 수동: 사람이 체크박스 클릭, 다음 태스크 수동 지시
- `/dependency-auto-executor`: 병렬 가능한 건 병렬, 아닌 건 대기 — 사람은 상태만 관찰
- **장점**: 4장 "오케스트레이터 역할"의 구체적 예

### 4.5 /pre-completion-verifier — "완료" 선언 뒤에 붙는 증거

- AI가 "완성" → 실제 빌드 실패 (2장 AP-10 재현)
- `/pre-completion-verifier` 활성 시 빌드·타입·테스트 결과를 **증거 블록**으로 붙여 옴

### 4.6 /systematic-debugger — 수정 전에 원인부터

- "버그야, 고쳐줘" → AI가 즉시 수정 시도 → 다른 곳이 깨짐
- `/systematic-debugger`: 재현 → 가설 → 검증 → 수정 순서

### 4.7 /auto-security-audit — 인증 붙는 순간 한 번

- Supabase auth 붙이자마자 한 번 돌림
- 리포트: RLS 정책 누락, 세션 토큰 노출 경로 등
- **장점**: 사람이 빠뜨리기 쉬운 지점을 자율 탐색이 잡아줌

### 4.8 /self-improvement-loop — 일회성 탈출

- 검색 정확도를 Frozen Metric으로 고정 (정답 라벨 50건)
- 인덱싱 전략을 3~5회 변형 → 자동 측정 → 점수 오른 버전만 keep
- **장점**: "한 번 만들고 끝"이 아니라 **측정 가능한 개선 루프**

### 4.9 /token-compressor, /cost-model-router — 공통 축소층

- 같은 파이프라인을 켜고/끄고 각각 실행
- 토큰 사용량·비용 차이를 한 장 테이블로

## 5. 발표 구간별 소비 계획

| 발표 장 | 예제에서 뽑아 쓰는 소재 | 형식 |
|---|---|---|
| 2장 (실패) | 4.1 Before / 4.5 "완료 선언 후 빌드 실패" + **카드 2** (hidden-assumption-surfacer) | 정적 스크린샷 2장 + 카드 1장 |
| 5장 (운영 방식) | 4.1 → 4.3 → 4.4 → 4.5 **한 흐름 편집** + **카드 1·3·5·6** 영상 직전 삽입 | 2~3분 사전 녹화 영상 + 상세 카드 4장 |
| 6장 (결함→해결 매핑) | 4.1/4.2/4.5/4.9/4.8을 5가지 결함에 1:1 매핑 + **카드 4** (task-decomposer) | 표 한 장 + 카드 1장 |
| 7장 (설정과 사용) | 실제 설정 파일 골격 — CLAUDE.md, settings.json의 `mcpServers`, `~/.claude/skills/` 구조 | 코드 스니펫 |
| 8장 (효과) | 4.9의 토큰·비용 대비 표, 4.8의 개선 곡선 + **카드 7** (self-improvement-loop) | 정적 표/차트 + 카드 1장 |
| 9장 (시작하기) | **본 시연에서 썼던 4개**(`/requirements-dialogue`, `/screen-spec-generator`, `/dependency-auto-executor`, `/pre-completion-verifier`)를 "시연 파이프라인의 최소 단위"로 짧게 재언급. **deck `9-1`의 도입 순서 5개와는 역할이 다름** — deck 9-1은 "가장 안전한 개인 도입 순서"(`/systematic-debugger`부터). 섞어서 말하지 않는다. | 도입 순서 다이어그램 |

> **카드 문서**: 각 스킬의 핵심 개념·구조·축약 코드는 `SKILLS-DEMO-CARDS.md`에 분리 정리. 영상 구간 직전에 1장씩 삽입한다(카드→영상→카드→영상 리듬). 카드 러닝 타임 매핑은 해당 문서 말미 참조.

## 7. 주의사항 (계획 단계에서 못 박아둘 것)

- **공개 범위 엄수**: `PRESENTATION-CONTEXT.md` 5절. 외부 저장소 링크 없음. 예제 코드도 외부 공개 금지. 슬라이드에 들어가는 스니펫은 경로·토큰·개인 경로 마스킹.
- **정밀 수치 금지**: 4.9의 토큰/비용 대비는 "대략 N% 절감" 범위로만. 98.5% 같은 정밀 수치는 백업 슬라이드로만 보유.
- **라이브 시연 배제**: 전부 사전 녹화. 영상 재생 실패 대비로 핵심 스크린샷 3장을 백업 슬라이드에 둠.
- **스킬명 표기**: 발표 슬라이드는 직관 이름(kebab-case), 이 계획서도 직관 이름 사용. 실제 실행 시 `SKILL-NAME-MAPPING.md`의 실제 스킬로 변환.
- **"스킬 자랑" 방지**: 각 장면은 "장점 증명"이 아니라 "결함 제거"로 프레이밍. 2/6장의 결함 담론과 톤을 맞춘다.

## 8. 완료 기준

아래 5개 중 4개 이상이면 예제 준비 완료로 간주한다.

1. 5장 영상에 `/requirements-dialogue → /screen-spec-generator → /dependency-auto-executor → /pre-completion-verifier` 흐름이 2~3분 편집본으로 들어가 있다
2. 6장 표의 5가지 결함 각각에 예제 스크린샷이 1장씩 매칭된다
3. 7장 설정 스니펫 6~7장이 모두 마스킹을 통과하고 실제 디렉토리·파일과 일치한다
4. 8장 토큰/비용 대비 표가 실측값 기반으로 하나 이상 존재한다 (백업 슬라이드 포함)
5. `SKILLS-DEMO-CARDS.md` 상세 카드 7장 + 묶음 카드 5장이 발표용으로 슬라이드화되어 있고, 카드 말미 공개 범위 체크리스트 5개 항목을 모두 통과한다
