# LLM Tools, Skills, and AI-Native Workflows

# 5장. 운영 방식

### 5-1. 핵심 원칙 3가지

**슬라이드**
```
계획은 함께 작성한다
   AI에게 "만들어줘"가 아니라 "같이 기획하자"

구현은 맡기되 그대로 믿지 않는다
    위임과 검증은 세트다

완료보다 증거를 우선한다
    "완성했습니다" < "테스트 통과 결과입니다"
```

*발표자 노트*
"제가 AI 에이전트를 쓰면서 정착한 원칙 세 가지입니다. 단순해 보이지만, 실제로 이걸 지키는 게 쉽지 않습니다. 특히 두 번째, '믿지 않는다'는 건 매번 모든 걸 의심하라는 게 아닙니다. 검증의 기준과 방법을 미리 정해놓는다는 의미입니다. 스킬을 사용해서 '완성했습니다'응답이 아니라 '어떤 테스트를 수행해서 통과했습니다'로 행동하게 해야합니다."

### 5-2. 사람과 에이전트의 역할 분담

**슬라이드**
```
사람이 맡는 것                    에이전트가 맡는 것
─────────────────────────────────────────────────────
목표와 기준 정하기                코드베이스 탐색
기획 방향 결정                    반복 구현 작업
완료 기준 설계                    테스트 실행
판단과 우선순위                   문서 초안
리뷰와 승인                      엣지케이스 탐색

핵심: 판단은 사람, 탐색과 생성은 에이전트
```

*발표자 노트*
"이 구분이 중요한 이유가 있습니다. 판단을 에이전트에게 넘기면 책임이 흐려집니다. 목표와 기준이 명확해야 에이전트도 제대로 일합니다. 반대로 탐색과 생성을 사람이 계속 직접 하면 AI를 도입한 의미가 없어집니다."


### 5-3. 세션 운영 원칙

**슬라이드**
```
세션을 유지하는 경우
  → 하나의 작업 단위가 끝나지 않았을 때
  → 맥락 연속성이 중요한 디버깅 중

세션을 끊는 경우
  → 작업 단위 하나가 완료됐을 때
  → 컨텍스트가 과도하게 쌓였을 때  (예: 100K+ 토큰)
  → 다음 작업이 이전 맥락을 필요로 하지 않을 때

세션 전환 시 핸드오프
  → 완료 내용, 남은 작업, 다음 진입 조건을 파일로 기록
```

*발표자 노트*
"세션을 언제 끊느냐가 처음엔 감이 안 잡힙니다. 제 기준은 '이 세션을 계속 이어가는 게 새로 시작하는 것보다 얼마나 이득인가'입니다. 맥락이 계속 쌓이면 비용도 늘고 환각도 늘어납니다. 작업 단위 하나가 끝나면 핸드오프 문서를 만들고 새 세션을 시작하는 게 대부분 더 효율적입니다. 세션을 하다가 업무가 길어지면 세션 내에서 수행하는 목표를 더 작게 나누고, 작은 목표를 하나씩 완료해가면서 세션을 유지하는게 더 좋습니다."


### 5-3b. 세션 유지 vs 갱신 — 각자의 장단점

**슬라이드**
```
세션 유지 (이어가기 / resume)          세션 갱신 (끊고 새로 시작)
──────────────────────────────────────────────────────────────
장점                                    장점
  맥락 연속성 — 앞선 결정·가정 유지        컨텍스트 클린 — 토큰 새로 시작
  핸드오프 비용 0                         잘못된 가정 리셋 기회
  빠른 재진입 (resume)                    비용 절감 효과 체감 큼

단점                                    단점
  컨텍스트 누적 → 앞부분 망각·환각          핸드오프 비용 (요약·진입 조건 기록)
  토큰 비용 선형 증가                      맥락 누락 위험
  초기 드리프트가 끝까지 이어짐             반복 설명이 필요할 수 있음

판단 기준
  "이어갈 때의 이득 > 새로 시작할 때의 비용?"
  작업 단위 닫히면 갱신, 맥락이 레버리지면 유지.
```

*발표자 노트*
"이분법이 아닙니다. 각자 뚜렷한 장단점이 있어서 상황에 따라 선택합니다. 세션 안에서 쌓인 계약·메트릭이 매 세션 재구성 비용보다 큰 경우도 있습니다. 반대로 한 세션에서 화면 하나 완성한 뒤엔 바로 끊습니다. 계속 이어가면 다음 화면까지 전 화면의 컨텍스트가 끌려오거든요. 판단 기준은 하나, '이어갈 이득이 새로 시작 비용보다 큰가' 입니다. 그리고 맥락을 유지할 필요가 있다면 세션을 길게 가져가는 것보다 중요 내용을 별도 문서로 분리하는 것이 중요합니다."

### 5-4. 토큰 운영 원칙

**슬라이드**
```
유지하는 맥락                    버리는 맥락
─────────────────────────────────────────────────
현재 작업 관련 파일               완료된 Phase 로그
현재 스펙                        중간 실험 결과
검증 기준                        불필요한 디렉토리 탐색 결과

토큰 절감 실전 (내 워크플로우 기준, 케이스별 편차 큼):
  git diff → git diff --stat       (상당 부분 감소)
  pytest 전체 → 마지막 20줄만       (큰 폭 감소, 출력량 의존)
  분석 에이전트는 읽기 도구만         (불필요한 쓰기 차단)
```

*발표자 노트*
"토큰을 아끼는 건 단순히 비용 문제가 아닙니다. 컨텍스트가 작을수록 AI가 현재 작업에 더 집중합니다. 제 워크플로우에서는 git diff 하나만 압축해도 눈에 띄게 줄었습니다. 다만 출력량이나 프로젝트 구조에 따라 편차가 크기 때문에, 직접 해보고 체감하는 게 숫자보다 정확합니다. 추상적으로 업무를 던질 때도 작은 단위로 업무를 요청하는 것과 큰 단위로 요청하는 거의 퀄리티가 달라집니다."


### 5-5. 완료 판정 기준

**슬라이드**
```
AI가 "완료"라고 할 때 확인하는 것:

  ① 빌드가 통과했는가?          (실행 결과 확인)
  ② 테스트가 존재하고 통과했는가? (커버리지 확인)
  ③ 스펙과 다른 부분은 없는가?   (명세 대조)
  ④ 에러 로그는 없는가?          (런타임 확인)

증거 없는 완료 주장 = 완료 아님
```

*발표자 노트*
"이게 처음엔 과하게 느껴질 수 있습니다. 매번 이걸 다 확인해야 하나? 하지만 이걸 체계화하면, 에이전트에게 '완료 전에 이것들을 확인하고 결과를 보여달라'고 지시할 수 있습니다. 사람이 일일이 하는 게 아니라 에이전트가 스스로 확인하게 만드는 게 목표입니다."

### 5-6. 실제 흐름: 사전 녹화 영상 (러닝시트)

**슬라이드**
```
실제 운영 흐름 데모 — 카드 → 영상 → 카드 → 영상

타임라인 (약 2분 50초, 2~3분 범위)
  0:00  도입 — "4단계를 짧게 보겠습니다"
  0:50  카드 ① requirements-dialogue        → 영상 구간 ①
  1:25  카드 ③ dependency-auto-executor     → 영상 구간 ②
  2:00  카드 ④ pre-completion-verifier      → 영상 구간 ③
  2:35  마감 — "이 4개로 5가지 결함 중 4개를 덮습니다 → 6장"

각 카드는 이 장의 뒤이은 슬라이드로 교차 재생됩니다.
```

*발표자 노트*
"지금부터 영상과 카드를 교차로 보실 겁니다. 카드 한 장(약 15초)으로 '왜 이 스킬이 있는가'를 짚고, 영상 구간(약 20초)으로 '실제로 어떻게 동작하는가'를 보는 리듬입니다. 영상은 사전 녹화본이고, 각 카드는 같은 5장 안에서 한 장씩 이어집니다."

**슬라이드**
마스터 디시전 트리
```
시작
 │
 ├─ Q1. 지금 무엇이 있는가?
 │   │
 │   ├─ "아이디어만 있다" ──────────────────→ [A. 아이디어 단계]
 │   ├─ "기획 문서가 있다" (docs/planning/) ─→ [B. 명세 단계]
 │   ├─ "TASKS.md가 있다" ─────────────────→ [C. 실행 단계]
 │   ├─ "기존 코드가 있다" (레거시) ─────────→ [D. 역공학 단계]
 │   └─ "아무것도 없다 / 잘 모르겠다" ──────→ [E. 탐색 단계]
 │
 ├─ Q2. 지금 무엇을 하려는가?
 │   │
 │   ├─ "품질을 검증하고 싶다" ────────────→ [F. 검증 단계]
 │   ├─ "버그를 고치고 싶다" ──────────────→ [G. 디버깅 단계]
 │   ├─ "코드를 개선하고 싶다" ────────────→ [H. 개선 단계]
 │   ├─ "비용/토큰을 줄이고 싶다" ─────────→ [I. 최적화 단계]
 │   └─ "새 프로젝트를 셋업하고 싶다" ─────→ [J. 부트스트랩 단계]

 └─ Q3. 어떤 기술을 쓰고 있는가?
     │
     └─ "특정 언어/프레임워크 전문가가 필요하다" → [K. 전문가 단계]
```

### 5-6a. 카드 ① + 영상 구간 ① — requirements-dialogue

**슬라이드**
```
requirements-dialogue                        [기획 · 1단계]
핵심: 10개 명확성 차원을 질문으로 되돌려 사용자가 결정권을 되찾게 함

축약 코드
  ---
  name: requirements-dialogue
  description: 1:1 대화로 요구사항 수렴 → 9개 기획 문서
  ---
  # Phase 0.5~1 (축약)
  #   존재론적 탐색 → 경쟁 분석 → 페르소나 → 비즈니스 모델
  #   Ambiguity ≤ 0.4 될 때까지 코드 X

Before  "주간 노트 허브 만들어줘"  →  AI가 Firebase + Clerk로 시작
After   10개 차원 질문  →  사용자가 Supabase + 이메일 OTP 선택

[영상 구간 ① 재생: 0:45~1:10]
```

*발표자 노트*
"첫 장면입니다. Before에서 AI가 질문 없이 달려나가는 걸 한 번 보여주고, 스킬을 켠 뒤엔 AI가 오히려 저한테 질문을 쏟아냅니다. 결정권이 저한테 돌아온 겁니다."


### 5-6c. 카드 ② + 영상 구간 ② — dependency-auto-executor

**슬라이드**
```
dependency-auto-executor                      [실행 · 5단계]
핵심: Worktree + TDD 강제 → Phase 리뷰/QA/보안 게이트 → 병합·push까지 한 루프

표준 실행 체인
  태스크 선택 → 전문가 위임 → Worktree 준비 → TDD(RED→GREEN→REFACTOR)
  → 자체 검증 + 로컬 커밋 → Phase 게이트 → main 병합 → 원격 push

핵심 규칙
  - 병합/push는 오케스트레이터만 (전문가는 구현·검증까지)
  - 병렬은 같은 Phase × 다른 파일 × 최대 3~4개
  - TASK_DONE:{task_id}:{commit_sha} 로만 완료 보고

Before  사람이 체크박스 찍으며 다음 태스크 수동 지시
After   Worktree 3개 병렬 동시 진행 → 사람은 Phase 게이트만 승인

[영상 구간 ③ 재생: 2:20~2:50]
```

*발표자 노트*
"여기서부터 4장에서 말씀드린 '오케스트레이터 역할'이 눈에 보입니다. Worktree 세 개가 동시에 돌고, 저는 Phase 게이트에서 승인만 누릅니다. 단, 병합과 push는 사람이 항상 확인합니다."

### 5-6d. 카드 ③ + 영상 구간 ③ — pre-completion-verifier

**슬라이드**
```
pre-completion-verifier                       [검증 · 품질 체인 1번]
핵심: "완료" 주장은 증거로 대체된다. 빌드·타입·테스트 실제 실행 후만 주장

Iron Law
  새로운 검증 증거 없이 완료 주장 금지

Gate Function (5단계)
  ① IDENTIFY  이 주장을 증명하는 명령은 무엇인가
  ② RUN       fresh, complete 실행
  ③ READ      전체 출력 + exit code + 실패 수
  ④ VERIFY    출력이 주장을 확인하는가
  ⑤ ONLY THEN 증거와 함께 주장

Before  AI "완료했습니다"  → 실제로는 타입 에러
After   AI가 npm run build / npm test 실행 → 마지막 20줄을 증거 블록으로 첨부

[영상 구간 ④ 재생: 3:10~3:30]
```

*발표자 노트*
"마지막 장면. 2장의 '증거 없는 완료' 장면을 이 스킬이 어떻게 덮는지 보실 겁니다. AI가 스스로 명령어를 실행하고, 출력 마지막 부분을 증거로 붙여옵니다. 이게 있으면 '완료라고 했는데 빌드는 안 되는' 상황이 구조적으로 사라집니다. 이 흐름 전체를 스킬 묶음이 구성합니다 — 6장에서 그 체계를 보겠습니다."
