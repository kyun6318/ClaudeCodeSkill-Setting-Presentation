# LLM Tools, Skills, and AI-Native Workflows

# 3장. 기술 요소의 실무적 의미

### 3-1. 계층 구조: Tool Calling → MCP → Skill

**슬라이드**
```
Skill          사용자의 운영 원칙을 고정하는 장치
   ↑              requirements-dialogue, dependency-auto-executor 등
MCP            외부 시스템을 모델에 연결하는 규약
   ↑              GitHub MCP, Notion MCP 등
Tool Calling   모델이 행동하는 수단
               파일 읽기, 코드 실행, 검색 등

각 계층은 역할이 다르다.
```

*발표자 노트*
"Tool Calling은 모델이 파일을 읽거나 코드를 실행하는 등 실제 행동을 하는 방식입니다. MCP는 그 도구들을 표준화된 방식으로 연결하는 규약이고요. Skill은 그보다 한 층 위에서, 이 도구들을 어떤 순서로 어떤 원칙으로 사용할지를 정의합니다. 7장에서 실제 설정 방법을 볼 때 이 계층 관계를 기억하시면 이해하기 쉬울 겁니다."

### 3-2. MCP vs API

**슬라이드**
```
API                              MCP
─────────────────────────────────────────────────
기능 인터페이스                  모델 연결 인터페이스
"이 기능을 호출하라"             "이 도구를 모델이 쓸 수 있게 연결"
코드에서 명시적으로 호출         모델이 필요할 때 자율적으로 선택
REST, GraphQL                    표준화된 도구 명세 (JSON Schema)

→ 경쟁 관계가 아니다. 계층이 다르다.
  API가 기능을 제공하고, MCP가 그 기능을 모델에 연결한다.
```

*발표자 노트*
"MCP가 나오고 나서 'API가 사라지는 건가?'라는 질문을 많이 받습니다. 아닙니다. API는 여전히 기능 인터페이스이고, MCP는 그 API를 모델이 쓸 수 있게 연결하는 레이어입니다. 그리고 MCP는 역시 확률적으로 동작합니다. 반면 API를 툴로 제공하면 확정된 값을 전달할 수 있습니다. 예를 들어 MCP로 DB를 연결하고 'DB에서 나이 많은 사람을 찾아줘'라고 요청하면 MCP로 연결된 디비에서 확률적으로 생성된 SQL이 실행되고 결과가 나옵니다. 어떤 테이블에서 어떤 값을 기준으로 값을 반환하는지 확정할 수 없습니다. 반면 API툴은 저희가 정해놓은 코드로만 답을 내놓습니다."

### 3-3. Agent Team과 책임 분리

**슬라이드**
```
Agent Team을 구성하는 기준: 병렬화보다 책임 분리

  ✓ 나눌 때          — 역할이 충돌할 때 (만들기 vs 평가하기)
                      — 컨텍스트를 격리해야 할 때
                      — 독립적으로 실행 가능할 때

  ✗ 나누지 말 때     — 단순히 빠르게 하고 싶다는 이유만으로
                      — 결과를 조율할 더 복잡한 코드가 필요할 때

핵심: 같은 에이전트가 만들고 평가하면 자기 코드에 관대해진다.
     만드는 에이전트와 평가하는 에이전트를 분리해야 한다.
```

*발표자 노트*
"Agent Team 패턴에서 가장 중요한 건 속도가 아니라 책임 분리입니다. 만드는 에이전트와 평가하는 에이전트를 분리하면, 평가자가 작성자의 자신감에 오염되지 않습니다. 이게 품질을 높이는 구조적 방법입니다."

### 3-4. 왜 Graph RAG만으로는 부족한가

**슬라이드**
```
Graph RAG: 데이터를 구조화하여 더 정확한 답변을 기대

한계:
  "완벽한 데이터를 줘도 출력은 여전히 확률적이다"
  "운영 문제는 데이터 품질로 해결되지 않는다"
  "같은 데이터, 같은 프롬프트 → 다른 결과가 나올 수 있다"

→ 데이터 문제를 푼다고 일관성 문제가 풀리지 않는다.
  운영 체계가 별도로 필요하다.

⇒ 운영 체계가 필요하다는 것은, 결국 사람의 역할 자체가 바뀐다는 뜻이다.
   (다음 장: 작업자 → 조율자)
```

*발표자 노트*
"Graph RAG는 강력한 기술입니다. 하지만 그것만으로 '에이전트가 항상 올바르게 작동한다'는 보장이 되지 않습니다. 출력이 확률적이고, 같은 맥락에서도 다른 결과가 나올 수 있습니다. 데이터 문제를 풀어도 운영 문제는 남습니다. 그리고 운영 문제는 결국 사람이 어떻게 일하느냐의 문제로 이어집니다. 다음 장에서 그 역할 변화를 봅니다."
