# LLM Tools, Skills, and AI-Native Workflows

# 2장. 운영 정책이 모호할 때 생기는 일

### 2-1. 실패 사례 1: 모호한 위임

**슬라이드**
```
"로그인 기능 만들어줘"

→ AI: [코딩 시작]
→ 완성: Google OAuth + JWT + Redis 세션 관리
→ 요청자 의도: 이메일/비밀번호 로그인이면 충분했음
```

*발표자 노트*
"질문이 모호하면 AI는 자기 나름대로 매번 다른 합리적인 선택을 합니다. 그 선택이 당신의 의도와 다를 확률이 꽤 높습니다. 문제는 이게 에이전트가 일을 많이 한 후에야 발견된다는 겁니다. 컨텍스트는 소비됐고, 시간도 썼고, 처음부터 다시 해야 합니다."

### 2-2. 실패 사례 2: 증거 없는 완료 선언

**슬라이드**
```
AI: "구현 완료했습니다."

실제 상태:
  ✗ 빌드 미확인
  ✗ 테스트 없음
  ✗ 명세와 다른 부분 있음
  ✓ 코드가 그럴듯하게 존재함

→ 사용자가 직접 확인이 필수
```

*발표자 노트*
"AI는 자신감 있게 틀립니다. '완성했습니다'라고 하면 우리는 확인 없이 믿고 싶어집니다. 그런데 빌드를 돌려보면 깨져있는 경우가 적지 않습니다. 에러가 없어도 스펙과 다르게 만들어놓은 경우도 있습니다. 이게 단순 실수가 아니라 구조적 문제라는 걸 인식하는 게 중요합니다."

### 2-2b. 카드: hidden-assumption-surfacer — "숨겨진 가정"의 구조

**슬라이드**
```
hidden-assumption-surfacer                        [메타 · 기획 보조]
핵심: AI가 대화 중 쌓는 가정을 4유형으로 분류 → hidden-assumption-surfacer.md로 가시화

구조                          축약 코드
─────────────────             ─────────────────────────────────
파이프라인: 메타               ---
트리거: 세션 어디서든          name: hidden-assumption-surfacer
출력: hidden-assumption-surfacer.md         disable-model-invocation: true
실제 스킬명: hidden-assumption-surfacer     ---
                               # 가정 분류 (축약)
                               #   [stated]    사용자가 명시
                               #   [inferred]  대화에서 추론
                               #   [assumed]   AI가 임의로 깐 것  ← 위험
                               #   [uncertain] 불확실, 질문 필요
                               #
                               # 신뢰 등급: ESTABLISHED > WORKING > OPEN

Before:  같은 스펙을 두 번 넣었는데 결과가 다름
After:   [assumed] 항목 5개가 먼저 쏟아져 나옴 → 사용자가 승격/기각
```

*발표자 노트*
"이 결함이 왜 '구조적'이냐 — AI는 매번 새로운 가정을 깔고 시작합니다. 이 스킬은 그 가정들을 4유형으로 분류해서 파일로 뽑아내는 것뿐인데, 그것만으로도 드리프트가 사전 차단됩니다. 7장에서 설정 구조가 다시 나오니 여기서는 '이런 장치가 있다' 정도만 잡고 넘어갑니다."

### 2-3. 세션과 토큰이 만드는 제약

**슬라이드**
```
세션: 대화의 '작업 공간'
  → 세션이 시작되면 새 창이 열린다
  → 이전 대화는 공유되지 않는다

토큰: 세션이 읽고 쓸 수 있는 '양'
  → 컨텍스트가 쌓일수록 토큰이 소비된다
  → 한계를 넘으면 앞부분을 잊거나 환각이 시작된다

세션이 길어질수록:
  ① 에러가 누적된다
  ② 초기 지시를 잊는다
  ③ 비용이 커진다
```

*발표자 노트*
"세션과 토큰은 AI 에이전트를 운영할 때 반드시 알아야 하는 물리적 제약입니다. 이걸 모르고 쓰면, 긴 작업을 하나의 세션에 몰아넣다가 나중에 AI가 앞부분 요구사항을 잊어버리거나 엉뚱한 결과를 냅니다. 더 긴 컨텍스트가 항상 좋은 게 아닙니다. 적절하게 나누는 설계가 필요합니다. 세션이 길어지면 클로드 코드는 알아서 압축을 하는데 이 압축의 성능이 신뢰성이 높지 않습니다."

### 2-4. 같은 요청, 다른 결과

**슬라이드**
```
"로그인 기능 만들어줘"

  월요일 → JWT + Redis 세션
  화요일 → 쿠키 기반 세션
  수요일 → NextAuth 라이브러리 추천

AI의 출력은 확률적이다.
동일 입력이 동일 출력을 보장하지 않는다.

→ "운이 좋았던 결과"와 "운영 체계로 만든 결과"는 다르다
```

*발표자 노트*
"세션, 토큰 문제에 더해 더 근본적인 특성이 있습니다. AI의 출력 자체가 확률적입니다. 같은 프롬프트를 다른 날 실행하면 다른 결과가 나옵니다. 한 번 잘 됐다고 해서 다음에도 잘 된다는 보장이 없습니다. 이게 단순한 불편함이 아니라, 운영 체계가 필요한 근본 이유 중 하나입니다."

### 2-5. 5가지 구조적 결함

**슬라이드**
```
AI 에이전트의 구조적 결함
(모델의 버그가 아니라, LLM이 동작하는 방식 자체에서 나온다)

  ① 기획 부재       — "만들어줘" → 요구사항 불일치 → 3배 재작업
  ② 숨겨진 가정     — AI가 암묵적으로 결정 → 의도와 다른 아키텍처
  ③ 검증 없는 완료  — "완성" 선언 → 실제론 빌드 실패
  ④ 컨텍스트 폭발   — 작업이 쌓이면 → 토큰 한계 → 환각
  ⑤ 일회성 생성     — 한 번 만들면 끝 → 엣지케이스 누락

이 결함들에 대한 대응이 5장, 6장, 7장에서 나옵니다.
```

*발표자 노트*
"정리하면 5가지입니다. 왜 '구조적'이냐면 — LLM은 프로젝트를 완수하도록 훈련된 게 아니라 다음 토큰을 예측하도록 훈련됩니다. 컨텍스트 윈도우는 물리적 한계가 있고, 출력은 확률적이며, 세션 간 기억이 없습니다. 모델이 아무리 좋아져도 이 성격은 정도의 차이만 있을 뿐 사라지지 않습니다. 특정 도구의 문제가 아니라는 겁니다. 이걸 알고 대응하는 게 운영 체계를 갖추는 이유입니다."
