# LLM Tools, Skills, and AI-Native Workflows

# 4장. AI 도입은 업무 방식도 바꾸게 만든다

### 4-1. 기존 업무 방식의 강점

**슬라이드**
```
한 가지 일을 깊게 파고드는 방식 + 작은 것을 쌓아 올라가는 방식

강점:
  → 맥락을 오래 유지하면 깊이 있는 결과가 나온다
  → 복잡한 판단이 필요한 작업에 최적
  → 집중도 높을 때 창의적 돌파가 일어난다

이 방식은 사라지지 않는다.
"깊은 사고 구간"은 사람이 맡아야 하는 영역이다.
```

*발표자 노트*
"AI가 도입되면 기존에 잘 하던 방식을 버려야 하는 것처럼 느껴질 수 있는데, 그렇지 않습니다. 한 가지 일에 집중해서 깊이 파고드는 능력은 AI가 대체하기 어려운 영역입니다. 오히려 이 능력을 AI를 통해 더 효율적으로 쓸 수 있게 됩니다. 작은 것을 쌓아올린다는 개념도 변화하게 됩니다. 사람이 작성할 때는 10000을 만들기 위해 1씩 쌓아가면서 조정했다면, 라이브러리나 프레임워크를 통해 1000을 기본값에 10씩 쌓아올라갔다면, Agent를 활용하면 서로 다른 모양의 100개 묶음 100개를 사용처에 맞게 깍아가며 합치는 과정에 가깝습니다."

### 4-2. AI 도입 후 사람의 역할이 달라진다

**슬라이드**
```
Before                           After
───────────────────────────────────────────────────
직접 탐색 → 직접 구현 → 직접 검토   조율 → 위임 → 검증 → 판단

"내가 다 한다"                    "내가 설계하고, 에이전트가 실행하고,
                                  내가 기준으로 판단한다"

새로 생기는 역할: 오케스트레이터
```

*발표자 노트*
"AI 도입 후에 가장 많이 바뀌는 건 역할입니다. 이전엔 내가 탐색하고 구현하고 검토했다면, 이후엔 에이전트를 어떻게 조율할지 설계하고, 결과를 어떤 기준으로 판단할지에 에너지를 씁니다. 이걸 '작업자에서 오케스트레이터로의 전환'이라고 부릅니다. 처음엔 어색한 전환입니다. '엉뚱한 작업으로 인해 손해보는 구간을 생각하면 내가 직접 하는 게 더 빠른데'라고 느끼는 구간이 있습니다. 그 구간을 어떻게 넘기느냐가 실제로 AI 도구를 잘 쓰는지 못 쓰는지를 가릅니다."

### 4-3. 멀티로 일한다는 것의 실제 의미

**슬라이드**
```
오해: 사람이 동시에 여러 일을 깊게 한다

실제:
  에이전트 A — [기능 A 구현 중]    ← 에이전트가 실행
  에이전트 B — [기능 B 테스트 중]  ← 에이전트가 실행
  에이전트 C — [문서 작성 중]      ← 에이전트가 실행
  나         — [상태 확인 + 판단]   ← 내가 하는 일

멀티 = 병렬 실행은 에이전트가, 상태 관리와 판단은 사람이
```

*발표자 노트*
"멀티로 일한다는 말을 들으면 '사람이 동시에 여러 일을 해야 하는 건가'라고 오해하기 쉽습니다. 아닙니다. 병렬로 실행하는 건 에이전트이고, 사람은 각 에이전트의 상태를 관리하고 결과를 판단하는 역할입니다. 이게 잘 작동하려면 '이 작업은 에이전트에게 위임 가능한가', '어디서부터 사람이 판단해야 하는가'를 구분하는 감각이 필요합니다. 제가 추천하는 것은 여러 작업을 동시에 감독하는 것보다 최대한 관련된 작업 단위로 병렬 작업을 시키고 관리 감독을 하는 것입니다."

### 4-4. 컨텍스트 스위칭의 비용

**슬라이드**
```
사람의 컨텍스트 스위칭 비용

  어려운 전환: 깊은 사고 → 다른 깊은 사고
              (코드 설계 → 문서 작성 → 다시 코드 설계)

  관리 가능한 전환: 에이전트 상태 확인
              (지시 → 에이전트 실행 → 결과 검토)

→ "피해야 할 전환"과 "의도적으로 만들어야 할 전환"이 다르다
→ 세션을 나누고, 핸드오프를 설계하고, 완료 기준을 사전에 정하는 이유
```

*발표자 노트*
"컨텍스트 스위칭은 나쁜 게 아닙니다. 어떤 전환이냐가 중요합니다. 깊은 사고 중에 계속 끊기면 집중력이 깨지고 품질이 떨어집니다. 반면 에이전트 상태를 확인하거나 다음 지시를 내리는 전환은 상대적으로 비용이 낮습니다. 이 두 가지를 잘 설계하는 것이 AI 도입 후 업무 방식의 핵심입니다."

### 4-5. 이 변화가 요구하는 것

**슬라이드**
```
새롭게 생기는 질문들:

  "이 작업을 어떻게 나눌 것인가?"
  "어느 시점에 세션을 끊을 것인가?"
  "에이전트가 완료했다고 할 때 어떻게 검증할 것인가?"
  "결과에 어떤 기준을 적용할 것인가?"

→ 이 질문들에 대한 답이 운영 원칙이다
→ 다음 장에서 구체적인 방법을 본다
```

*발표자 노트*
"4장 전체를 요약하면, AI를 도입하면 이런 새로운 질문들이 생긴다는 겁니다. 이 질문들이 당황스러울 수 있는데, 이게 정상입니다. 운영 원칙이 없으면 이 질문들을 매번 다시 고민해야 합니다. 5장에서는 제가 실제로 어떻게 답하고 있는지 구체적으로 말씀드리겠습니다."
