### 1-0. 오늘 다루는 것 / 다루지 않는 것

**슬라이드**
```
오늘 다루는 것
  • Claude Code를 "어떻게 쓰고 설계하고 평가하는가"
  • 운영 원칙과 그 원칙을 반복 가능하게 만드는 장치(Skill)
  • 작업자 → 조율자로의 역할 변화

오늘 다루지 않는 것
  • LLM / 프롬프트 기초
  • AI 도입의 당위성 설득
  • 특정 툴의 기능 나열식 소개
```

*발표자 노트*
"본문 75분 + Q&A 15분으로 갑니다. 첫 슬라이드는 기대치 맞추기입니다. 오늘은 '뭐가 되는가'가 아니라 '어떻게 쓰는가'입니다. 기초 개념이나 도입 당위성은 이미 아시는 분이 많으니 건너뜁니다. '정답 공유'가 아니라 '쟤는 저렇게 쓰고 있구나' 정도로 들어주시면 됩니다."

### 1-1. 오프닝: 결론부터 — 왜 지금 '운영' 이야기인가

**슬라이드**
```
에이전트가 막 쓸 만해진 지금, 쓰는 사람마다 결과 편차가 크다.
차이는 모델이 아니라 운영에서 나온다.

① 에이전트는 사람을 대체하는 도구가 아니다
② 잘 쓰려면 운영 원칙이 필요하다
③ 스킬은 그 운영 원칙을 반복 가능하게 고정하는 장치다

─────────────────────────────────────────────
본 발표는 Claude Code 중심 사례.
다만 운영 원칙은 다른 에이전트 도구에도 일반화 가능.
```

*발표자 노트*
"모델은 누구나 같은 걸 씁니다. 그런데 결과는 사람마다 다릅니다. 그 차이가 모델이 아니라 **운영**에서 나온다는 게 오늘의 출발점입니다. 그래서 말하고 싶은 건 이 세 문장입니다.

첫째, 에이전트는 사람을 대체하지 않고 **역할 구성을 바꿉니다** — 작업자에서 조율자로. 4장에서 다룹니다.
둘째, 그냥 던지면 실패합니다. 왜 실패하는지는 2장에서 봅니다.
셋째, 스킬은 그 운영 원칙을 매번 다시 말하지 않아도 되게 **고정**하는 장치입니다.

마지막 줄은 프레이밍입니다. 제가 쓰는 Claude Code 기준이지만, 원칙은 다른 도구에도 적용됩니다."

### 1-2. 확률적 출력이라는 전제 — 베이즈 비유

**슬라이드**
```
같은 질문에 다른 답이 나오는 이유

  모델의 출력 = 분포에서 뽑는 표본 (0/1의 확정 아님)
    │
    ├─ 맥락 얇다 → 분포 넓다 → 답이 흔들린다
    ├─ 맥락 두껍다 → 분포 좁다 → 답이 일관된다
    └─ 운영 원칙 = 맥락을 "어떻게 두껍게 유지할지"

∴ 이 발표는 정답 찾기가 아니라, 분포를 좁히는 운영의 이야기다
```

*발표자 노트*
"한 문장만 기억해 주세요 — LLM 출력은 확정이 아니라 **분포에서의 표본**입니다. 0/1로 되느냐 안 되느냐의 사고를 먼저 내려놓으셔야 합니다. 맥락이 얇으면 분포가 넓어서 답이 흔들리고, 두껍게 유지하면 좁혀집니다. 오늘 이야기하는 세션 운영·스킬·핸드오프 — 전부 '맥락을 어떻게 두껍게 유지할지'입니다. 그래서 이 발표는 **정답 찾기가 아니라 분포 좁히기**에 대한 이야기입니다. 수식은 안 씁니다."

---

### 1-3. 앱 vs CLI vs IDE — 오늘의 범위는 CLI

**슬라이드**
| 구분 | Claude.ai 웹/앱 | **Claude Code (CLI)** | Cursor / Antigravity (IDE) |
|---|---|---|---|
| 로컬 파일·git·터미널 | ❌ | ✅ | ✅ |
| 세션·권한 위임 | 제한 | 세밀 제어 | 도구별 상이 |
| 확장 메커니즘 | Instructions, Projects | **Skills / MCP / Hooks / Sub-agent** | 유사 개념, 동일 메커니즘 아님 |

> 오늘 이야기는 전부 **CLI 기준**. 앱에서는 동작하지 않음.

*발표자 노트*
"이거 하나만 짚고 넘어가겠습니다. 오늘 말하는 Skill·MCP·오케스트레이션은 전부 **CLI 기준**이고 앱에서는 안 됩니다. 저도 앱 → Cursor → 지금 Claude Code로 옮겨왔고, 각 도구에 유사 개념은 있지만 동일 메커니즘은 아닙니다. 이 표 하나로 끝내고 바로 넘어가겠습니다."

### 1-4. 이 발표의 관점과 청자가 가져갈 것

**슬라이드**
```
관점
  장점 소개 ❌ / 운영 체계 공유 ✅
  "내가 어떤 기준으로 쓰고, 어떻게 설계하고 평가하는가"

청자가 가져갈 것
  1. 운영 감각 — 설정·흐름·주의점
  2. 평가 기준 — 내 체계가 잘 되고 있는지 보는 눈
  3. 시작점  — 어디부터, 어떤 순서로
  4. 재공유  — 직접 써보고 경험을 돌려주세요 (오늘의 CTA)
```

*발표자 노트*
"장점 자랑 발표가 아닙니다. 운영 체계를 공유하고, 그걸 **냉정하게 평가하는 방법**을 같이 보는 자리입니다. 마지막 줄이 오늘의 Call to Action입니다 — 직접 써보시고 각자 방식을 돌려주시는 게 저한테 가장 좋은 피드백입니다."

# 2장. 왜 그냥 맡기면 실패하는가

---

### 2-1. 실패 사례 1: 모호한 위임

**슬라이드**
```
"로그인 기능 만들어줘"

→ AI: [코딩 시작]
→ 완성: Google OAuth + JWT + Redis 세션 관리
→ 요청자 의도: 이메일/비밀번호 로그인이면 충분했음

다시 처음부터.
```

*발표자 노트*
"이런 경험 있으시죠? 질문이 모호하면 AI는 자기 나름대로 합리적인 선택을 합니다. 그 선택이 당신의 의도와 다를 확률이 꽤 높습니다. 문제는 이게 에이전트가 일을 많이 한 후에야 발견된다는 겁니다. 컨텍스트는 소비됐고, 시간도 썼고, 처음부터 다시 해야 합니다. 이걸 '3배 재작업'이라고 부릅니다."*

---

### 2-2. 실패 사례 2: 증거 없는 완료 선언

**슬라이드**
```
AI: "구현 완료했습니다."

실제 상태:
  ✗ 빌드 미확인
  ✗ 테스트 없음
  ✗ 명세와 다른 부분 있음
  ✓ 코드가 그럴듯하게 존재함

→ 사용자가 직접 확인하기 전까지 알 수 없다
```

*발표자 노트*
"AI는 자신감 있게 틀립니다. '완성했습니다'라고 하면 우리는 확인 없이 믿고 싶어집니다. 그런데 빌드를 돌려보면 깨져있는 경우가 적지 않습니다. 에러가 없어도 스펙과 다르게 만들어놓은 경우도 있습니다. 이게 단순 실수가 아니라 구조적 문제라는 걸 인식하는 게 중요합니다."*

---

### 2-3. 세션과 토큰이 만드는 제약

**슬라이드**
```
세션: 대화의 '작업 공간'
  → 세션이 시작되면 새 창이 열린다
  → 이전 대화는 공유되지 않는다

토큰: 세션이 읽고 쓸 수 있는 '양'
  → 컨텍스트가 쌓일수록 토큰이 소비된다
  → 한계를 넘으면 앞부분을 잊거나 환각이 시작된다

세션이 길어질수록:
  ① 에러가 누적된다
  ② 초기 지시를 잊는다
  ③ 비용이 커진다
```

*발표자 노트*
"세션과 토큰은 AI 에이전트를 운영할 때 반드시 알아야 하는 물리적 제약입니다. 이걸 모르고 쓰면, 긴 작업을 하나의 세션에 몰아넣다가 나중에 AI가 앞부분 요구사항을 잊어버리거나 엉뚱한 결과를 냅니다. 더 긴 컨텍스트가 항상 좋은 게 아닙니다. 적절하게 나누는 설계가 필요합니다."*

---

### 2-4. 같은 요청, 다른 결과

**슬라이드**
```
"로그인 기능 만들어줘"

  월요일 → JWT + Redis 세션
  화요일 → 쿠키 기반 세션
  수요일 → NextAuth 라이브러리 추천

AI의 출력은 확률적이다.
동일 입력이 동일 출력을 보장하지 않는다.

→ "운이 좋았던 결과"와 "운영 체계로 만든 결과"는 다르다
```

*발표자 노트*
"세션, 토큰 문제에 더해 더 근본적인 특성이 있습니다. AI의 출력 자체가 확률적입니다. 같은 프롬프트를 다른 날 실행하면 다른 결과가 나옵니다. 한 번 잘 됐다고 해서 다음에도 잘 된다는 보장이 없습니다. 이게 단순한 불편함이 아니라, 운영 체계가 필요한 근본 이유 중 하나입니다."*

---

### 2-5. 5가지 구조적 결함

**슬라이드**
```
AI 에이전트의 구조적 결함
(모델의 버그가 아니라, LLM이 동작하는 방식 자체에서 나온다)

  ① 기획 부재       — "만들어줘" → 요구사항 불일치 → 3배 재작업
  ② 숨겨진 가정     — AI가 암묵적으로 결정 → 의도와 다른 아키텍처
  ③ 검증 없는 완료  — "완성" 선언 → 실제론 빌드 실패
  ④ 컨텍스트 폭발   — 작업이 쌓이면 → 토큰 한계 → 환각
  ⑤ 일회성 생성     — 한 번 만들면 끝 → 엣지케이스 누락

이 결함들에 대한 대응이 5장, 6장, 7장에서 나옵니다.
```

*발표자 노트*
"정리하면 5가지입니다. 왜 '구조적'이냐면 — LLM은 프로젝트를 완수하도록 훈련된 게 아니라 다음 토큰을 예측하도록 훈련됩니다. 컨텍스트 윈도우는 물리적 한계가 있고, 출력은 확률적이며, 세션 간 기억이 없습니다. 모델이 아무리 좋아져도 이 성격은 정도의 차이만 있을 뿐 사라지지 않습니다. 특정 도구의 문제가 아니라는 겁니다. 이걸 알고 대응하는 게 운영 체계를 갖추는 이유입니다."*

# 3장. 기술 요소의 실무적 의미

---

### 3-1. 계층 구조: Tool Calling → MCP → Skill

**슬라이드**
```
Skill          사용자의 운영 원칙을 고정하는 장치
   ↑              /requirements-dialogue, /dependency-auto-executor 등
MCP            외부 시스템을 모델에 연결하는 규약
   ↑              GitHub MCP, Notion MCP 등
Tool Calling   모델이 행동하는 수단
               파일 읽기, 코드 실행, 검색 등

각 계층은 역할이 다르다.
```

*발표자 노트*
"Tool Calling은 모델이 파일을 읽거나 코드를 실행하는 등 실제 행동을 하는 방식입니다. MCP는 그 도구들을 표준화된 방식으로 연결하는 규약이고요. Skill은 그보다 한 층 위에서, 이 도구들을 어떤 순서로 어떤 원칙으로 사용할지를 정의합니다. 7장에서 실제 설정 방법을 볼 때 이 계층 관계를 기억하시면 이해하기 쉬울 겁니다."*

---

### 3-2. MCP vs API

**슬라이드**
```
API                              MCP
─────────────────────────────────────────────────
기능 인터페이스                  모델 연결 인터페이스
"이 기능을 호출하라"             "이 도구를 모델이 쓸 수 있게 연결"
코드에서 명시적으로 호출         모델이 필요할 때 자율적으로 선택
REST, GraphQL                    표준화된 도구 명세 (JSON Schema)

→ 경쟁 관계가 아니다. 계층이 다르다.
  API가 기능을 제공하고, MCP가 그 기능을 모델에 연결한다.
```

*발표자 노트*
"MCP가 나오고 나서 'API가 사라지는 건가?'라는 질문을 많이 받습니다. 아닙니다. API는 여전히 기능 인터페이스이고, MCP는 그 API를 모델이 쓸 수 있게 연결하는 레이어입니다."*

---

### 3-3. Sub Agent와 책임 분리

**슬라이드**
```
Sub Agent를 나누는 기준: 병렬화보다 책임 분리

  ✓ 나눌 때          — 역할이 충돌할 때 (만들기 vs 평가하기)
                      — 컨텍스트를 격리해야 할 때
                      — 독립적으로 실행 가능할 때

  ✗ 나누지 말 때     — 단순히 빠르게 하고 싶다는 이유만으로
                      — 결과를 조율할 더 복잡한 코드가 필요할 때

핵심: 같은 에이전트가 만들고 평가하면 자기 코드에 관대해진다.
     만드는 에이전트와 평가하는 에이전트를 분리해야 한다.
```

*발표자 노트*
"Sub Agent 패턴에서 가장 중요한 건 속도가 아니라 책임 분리입니다. 만드는 에이전트와 평가하는 에이전트를 분리하면, 평가자가 작성자의 자신감에 오염되지 않습니다. 이게 품질을 높이는 구조적 방법입니다."*

---

### 3-4. 왜 Graph RAG만으로는 부족한가

**슬라이드**
```
Graph RAG: 데이터를 구조화하여 더 정확한 답변을 기대

한계:
  "완벽한 데이터를 줘도 출력은 여전히 확률적이다"
  "운영 문제는 데이터 품질로 해결되지 않는다"
  "같은 데이터, 같은 프롬프트 → 다른 결과가 나올 수 있다"

→ 데이터 문제를 푼다고 일관성 문제가 풀리지 않는다.
  운영 체계가 별도로 필요하다.
```

*발표자 노트*
"Graph RAG는 강력한 기술입니다. 하지만 그것만으로 '에이전트가 항상 올바르게 작동한다'는 보장이 되지 않습니다. 출력이 확률적이고, 같은 맥락에서도 다른 결과가 나올 수 있습니다. 이게 4장으로 넘어가는 이유입니다. 데이터 문제를 풀어도 운영 문제는 남습니다."*

# 4장. AI 도입은 업무 방식도 바꾸게 만든다

---

### 4-1. 기존 업무 방식의 강점

**슬라이드**
```
한 가지 일을 깊게 파고드는 방식

강점:
  → 맥락을 오래 유지하면 깊이 있는 결과가 나온다
  → 복잡한 판단이 필요한 작업에 최적
  → 집중도 높을 때 창의적 돌파가 일어난다

이 방식은 사라지지 않는다.
"깊은 사고 구간"은 사람이 맡아야 하는 영역이다.
```

*발표자 노트*
"AI가 도입되면 기존에 잘 하던 방식을 버려야 하는 것처럼 느껴질 수 있는데, 그렇지 않습니다. 한 가지 일에 집중해서 깊이 파고드는 능력은 AI가 대체하기 어려운 영역입니다. 오히려 이 능력을 AI를 통해 더 효율적으로 쓸 수 있게 됩니다."*

---

### 4-2. AI 도입 후 사람의 역할이 달라진다

**슬라이드**
```
Before                           After
───────────────────────────────────────────────────
직접 탐색 → 직접 구현 → 직접 검토   조율 → 위임 → 검증 → 판단

"내가 다 한다"                    "내가 설계하고, 에이전트가 실행하고,
                                  내가 기준으로 판단한다"

새로 생기는 역할: 오케스트레이터
```

*발표자 노트*
"AI 도입 후에 가장 많이 바뀌는 건 역할입니다. 이전엔 내가 탐색하고 구현하고 검토했다면, 이후엔 에이전트를 어떻게 조율할지 설계하고, 결과를 어떤 기준으로 판단할지에 에너지를 씁니다. 이걸 '작업자에서 오케스트레이터로의 전환'이라고 부릅니다. 처음엔 어색한 전환입니다. '내가 직접 하는 게 더 빠른데'라고 느끼는 구간이 있습니다. 그 구간을 어떻게 넘기느냐가 실제로 AI 도구를 잘 쓰는지 못 쓰는지를 가릅니다."*

---

### 4-3. 멀티로 일한다는 것의 실제 의미

**슬라이드**
```
오해: 사람이 동시에 여러 일을 깊게 한다

실제:
  에이전트 A — [기능 A 구현 중]    ← 에이전트가 실행
  에이전트 B — [기능 B 테스트 중]  ← 에이전트가 실행
  에이전트 C — [문서 작성 중]      ← 에이전트가 실행
  나         — [상태 확인 + 판단]   ← 내가 하는 일

멀티 = 병렬 실행은 에이전트가, 상태 관리와 판단은 사람이
```

*발표자 노트*
"멀티로 일한다는 말을 들으면 '사람이 동시에 여러 일을 해야 하는 건가'라고 오해하기 쉽습니다. 아닙니다. 병렬로 실행하는 건 에이전트이고, 사람은 각 에이전트의 상태를 관리하고 결과를 판단하는 역할입니다. 이게 잘 작동하려면 '이 작업은 에이전트에게 위임 가능한가', '어디서부터 사람이 판단해야 하는가'를 구분하는 감각이 필요합니다."*

---

### 4-4. 컨텍스트 스위칭의 비용

**슬라이드**
```
사람의 컨텍스트 스위칭 비용

  어려운 전환: 깊은 사고 → 다른 깊은 사고
              (코드 설계 → 문서 작성 → 다시 코드 설계)

  관리 가능한 전환: 에이전트 상태 확인
              (지시 → 에이전트 실행 → 결과 검토)

→ "피해야 할 전환"과 "의도적으로 만들어야 할 전환"이 다르다
→ 세션을 나누고, 핸드오프를 설계하고, 완료 기준을 사전에 정하는 이유
```

*발표자 노트*
"컨텍스트 스위칭은 나쁜 게 아닙니다. 어떤 전환이냐가 중요합니다. 깊은 사고 중에 계속 끊기면 집중력이 깨지고 품질이 떨어집니다. 반면 에이전트 상태를 확인하거나 다음 지시를 내리는 전환은 상대적으로 비용이 낮습니다. 이 두 가지를 잘 설계하는 것이 AI 도입 후 업무 방식의 핵심입니다."*

---

### 4-5. 이 변화가 요구하는 것

**슬라이드**
```
새롭게 생기는 질문들:

  "이 작업을 어떻게 나눌 것인가?"
  "어느 시점에 세션을 끊을 것인가?"
  "에이전트가 완료했다고 할 때 어떻게 검증할 것인가?"
  "결과에 어떤 기준을 적용할 것인가?"

→ 이 질문들에 대한 답이 운영 원칙이다
→ 다음 장에서 구체적인 방법을 본다
```

*발표자 노트*
"4장 전체를 요약하면, AI를 도입하면 이런 새로운 질문들이 생긴다는 겁니다. 이 질문들이 당황스러울 수 있는데, 이게 정상입니다. 운영 원칙이 없으면 이 질문들을 매번 다시 고민해야 합니다. 5장에서는 제가 실제로 어떻게 답하고 있는지 구체적으로 말씀드리겠습니다."*

# 5장. 내가 사용하는 운영 방식

---

### 5-1. 핵심 원칙 3가지

**슬라이드**
```
원칙 ①  계획은 함께 작성한다
         AI에게 "만들어줘"가 아니라 "같이 기획하자"

원칙 ②  구현은 맡기되 그대로 믿지 않는다
         위임과 검증은 세트다

원칙 ③  완료보다 증거를 우선한다
         "완성했습니다" < "테스트 통과 결과입니다"
```

*발표자 노트*
"제가 AI 에이전트를 쓰면서 정착한 원칙 세 가지입니다. 단순해 보이지만, 실제로 이걸 지키는 게 쉽지 않습니다. 특히 두 번째, '믿지 않는다'는 건 매번 모든 걸 의심하라는 게 아닙니다. 검증의 기준과 방법을 미리 정해놓는다는 의미입니다."*

---

### 5-2. 사람과 에이전트의 역할 분담

**슬라이드**
```
사람이 맡는 것                    에이전트가 맡는 것
─────────────────────────────────────────────────────
목표와 기준 정하기                코드베이스 탐색
기획 방향 결정                    반복 구현 작업
완료 기준 설계                    테스트 실행
판단과 우선순위                   문서 초안
리뷰와 승인                      엣지케이스 탐색

핵심: 판단은 사람, 탐색과 생성은 에이전트
```

*발표자 노트*
"이 구분이 중요한 이유가 있습니다. 판단을 에이전트에게 넘기면 책임이 흐려집니다. 목표와 기준이 명확해야 에이전트도 제대로 일합니다. 반대로 탐색과 생성을 사람이 계속 직접 하면 AI를 도입한 의미가 없어집니다."*

---

### 5-3. 세션 운영 원칙

**슬라이드**
```
세션을 유지하는 경우
  → 하나의 작업 단위가 끝나지 않았을 때
  → 맥락 연속성이 중요한 디버깅 중

세션을 끊는 경우
  → 작업 단위 하나가 완료됐을 때
  → 컨텍스트가 과도하게 쌓였을 때  (예: 100K+ 토큰)
  → 다음 작업이 이전 맥락을 필요로 하지 않을 때

세션 전환 시 핸드오프
  → 완료 내용, 남은 작업, 다음 진입 조건을 파일로 기록
```

*발표자 노트*
"세션을 언제 끊느냐가 처음엔 감이 안 잡힙니다. 제 기준은 '이 세션을 계속 이어가는 게 새로 시작하는 것보다 얼마나 이득인가'입니다. 맥락이 계속 쌓이면 비용도 늘고 환각도 늘어납니다. 작업 단위 하나가 끝나면 핸드오프 문서를 만들고 새 세션을 시작하는 게 대부분 더 효율적입니다."*

---

### 5-4. 토큰 운영 원칙

**슬라이드**
```
유지하는 맥락                    버리는 맥락
─────────────────────────────────────────────────
현재 작업 관련 파일               완료된 Phase 로그
현재 스펙                        중간 실험 결과
검증 기준                        불필요한 디렉토리 탐색 결과

토큰 절감 실전 (내 워크플로우 기준 추정, 케이스별 편차 큼):
  git diff → git diff --stat   (~80%)
  pytest 전체 → 마지막 20줄만   (~90%, 출력량에 따라 다름)
  분석 에이전트는 읽기 도구만    (불필요한 쓰기 차단)
```

*발표자 노트*
"토큰을 아끼는 건 단순히 비용 문제가 아닙니다. 컨텍스트가 작을수록 AI가 현재 작업에 더 집중합니다. 제 워크플로우에서는 git diff 하나만 압축해도 눈에 띄게 줄었습니다. 다만 출력량이나 프로젝트 구조에 따라 편차가 크기 때문에, 직접 해보고 체감하는 게 숫자보다 정확합니다."*

---

### 5-5. 완료 판정 기준

**슬라이드**
```
AI가 "완료"라고 할 때 확인하는 것:

  ① 빌드가 통과했는가?          (실행 결과 확인)
  ② 테스트가 존재하고 통과했는가? (커버리지 확인)
  ③ 스펙과 다른 부분은 없는가?   (명세 대조)
  ④ 에러 로그는 없는가?          (런타임 확인)

증거 없는 완료 주장 = 완료 아님
```

*발표자 노트*
"이게 처음엔 과하게 느껴질 수 있습니다. 매번 이걸 다 확인해야 하나? 하지만 이걸 체계화하면, 에이전트에게 '완료 전에 이것들을 확인하고 결과를 보여달라'고 지시할 수 있습니다. 사람이 일일이 하는 게 아니라 에이전트가 스스로 확인하게 만드는 게 목표입니다."*

---

### 5-6. 실제 흐름: 사전 녹화 영상

**슬라이드**
```
실제 운영 흐름 데모

  ① /requirements-dialogue — 기획: AI와 함께 요구사항 명확화
  ② /screen-spec-generator — 명세: 화면별 구현 스펙 생성
  ③ /dependency-auto-executor — 실행: 의존성 기반 자동 구현

[영상 재생: 2-3분]
```

*발표자 노트*
"제가 실제로 사용하는 흐름을 짧게 편집한 영상입니다. 보시면 이런 흐름이 됩니다.
[영상 재생]
이게 5장에서 이야기한 원칙들이 실제로 어떻게 적용되는지의 예시입니다. 이 흐름 전체를 스킬이 구성합니다. 6장에서 스킬 체계를 보겠습니다."*

# 6장. 스킬 묶음은 어떤 문제를 해결하는가

---

### 6-1. 5가지 결함 → 해결 매핑

**슬라이드**
```
결함                    해결 스킬 (예시)
────────────────────────────────────────────────────────
① 기획 부재           /requirements-dialogue, /screen-spec-generator, /task-decomposer
② 숨겨진 가정         /hidden-assumption-surfacer, /technical-premise-deconstructor
③ 검증 없는 완료      /pre-completion-verifier, /quality-gate
④ 컨텍스트 폭발       /token-compressor, /ultra-thin-executor
⑤ 일회성 생성         /self-improvement-loop (Frozen Metric + 자율 개선)

각 스킬은 특정 실패 방식에 대한 체계적 대응이다.
```

*발표자 노트*
"2장에서 말씀드린 5가지 결함, 기억하시죠? 65개 스킬은 그 결함들에 하나씩 대응하기 위해 만들어졌습니다. 무작위로 늘어난 도구 모음이 아니라, 구체적인 실패 경험에서 태어난 체계입니다."*

---

### 6-2. 65개 스킬 전체 맵 (한 장)

**슬라이드**
```
[전체 스킬 맵 이미지]

3가지 범주:
  ① 프롬프트 고정 (~40개)  — "매번 같은 품질 보장"
  ② 프로세스 강제 (~15개)  — "건너뛸 수 없게"
  ③ 진짜 새 능력 (~10개)   — "구조적으로만 가능한 것"

자세한 목록: GitHub 링크
```

*발표자 노트*
"65개 전부를 보려면 GitHub 코드를 보시면 됩니다. 여기선 세 가지 범주만 이해하시면 됩니다. 프롬프트 고정은 매번 500자짜리 지시를 '/requirements-dialogue' 한 마디로 줄이는 것, 프로세스 강제는 AI가 자발적으로 건너뛰는 단계를 반드시 거치게 만드는 것, 진짜 새 능력은 프롬프트만으로는 구조적으로 불가능한 것들입니다."*

---

### 6-3. 파이프라인 관점

**슬라이드**
```
아이디어 → 기획 → 명세 → 실행 → 검증 → 개선

  /idea-concretizer, /co-brainstorming         ← 아이디어 발산
  /requirements-dialogue, /auto-planner  ← 기획
  /screen-spec-generator              ← 화면 명세
  /task-decomposer          ← 태스크 분해
  /dependency-auto-executor         ← 실행
  /quality-gate, /code-reviewer ← 검증
  /self-improvement-loop             ← 자율 개선

각 단계를 스킬이 담당한다.
단계를 건너뛰면 그 단계의 결함이 나중에 나타난다.
```

*발표자 노트*
"파이프라인으로 보면 왜 이렇게 많은 스킬이 필요한지 이해하기 쉽습니다. 각 단계에는 고유한 실패 방식이 있고, 각 스킬은 그 단계의 실패를 방지합니다."*

---

### 6-4. 스킬의 수명

**슬라이드**
```
플랫폼이 흡수할 수 있는 스킬
  → 언어별 베스트 프랙티스 (python-pro 등)
  → 기본 검증 패턴 (verification 일부)
  → 토큰 최적화 (플랫폼 효율 개선 시)

사용자 고유성이 강한 스킬 (남는 것)
  → 사용자의 파이프라인 강제 (dependency-auto-executor, full-pipeline-orchestrator)
  → 사용자의 철학적 프레임 (requirements-dialogue, technical-premise-deconstructor)
  → 구조적 자가개선 (self-improvement-loop Frozen Metric)

→ "불필요해진다"가 아니라
   "플랫폼이 일부 흡수하지만, 고유한 것은 남는다"
```

*발표자 노트*

**슬라이드**

  → 필요한 스킬만 로드
  → 스킬별로 독립 수정 가능

### 8-1. 평가 프레임
