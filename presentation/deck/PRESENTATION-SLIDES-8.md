### 8-1. 평가 프레임

**슬라이드**
```
이 체계를 어떻게 평가하는가

관점        성공 기준                    한계 기준
──────────────────────────────────────────────────────
품질        재작업 감소, 검증 루프 정착   성숙도 편차 (일부 스킬 미완)
운영        세션 운영 안정, 핸드오프 정착  설정 복잡도, 학습 곡선
커버리지    기획→코드 파이프라인 완성     배포/운영/모니터링 영역 공백

평가 방식: 정성 경험 + 문서 성숙도 + ROI 추정 + 리스크 분석
```

*발표자 노트*
"이 체계를 냉정하게 평가한다면, 성공한 부분과 부족한 부분이 모두 있습니다. 기획부터 구현까지의 파이프라인은 안정적으로 작동합니다. 하지만 배포나 운영 단계는 아직 체계가 덜 갖춰져 있습니다. 이걸 솔직하게 말씀드리는 게 중요하다고 생각합니다."

### 8-1b. 카드: self-improvement-loop — "일회성 생성"을 막는 장치

**슬라이드**
```
self-improvement-loop                        [자율 개선 · 7단계]
핵심: Frozen Metric + keep/revert 루프. "한 번 만들고 끝"을 구조적으로 차단

3파일 원칙 (불변)                     2계층 RSI
──────────────────────────            ──────────────────────
prepare.*   고정 인프라 ❌ 수정 금지    L0 meta_eval   절대 불변
target.*    실험 대상 ✅ 에이전트만     L1   outer     전략 / program.md
program.md  목표·제약·힌트 (인간)      L1.5 eval      Frozen Metric
                                       L2   target    Inner Loop 영역

Verify vs Guard
  Verify  "메트릭이 개선되었나?"      → 숫자
  Guard   "다른 것이 깨졌나?"         → pass/fail
  Guard 실패 → revert → rework(최대 2회) → discard

실제 스킬명: self-improvement-loop
```

*발표자 노트*
"이 카드를 한 장 짚고 효과·한계로 넘어갑니다. 여기서 중요한 건 '평가 코드는 못 건드린다'입니다. 7장 안티패턴에서 '치명적 TOP 3'에 Frozen Metric 수정이 들어가는 이유가 이겁니다. 평가 기준을 낮추면 점수는 오르지만 실제론 나빠집니다."

### 8-2. 효과와 한계를 함께

**슬라이드**
```
효과                               한계
──────────────────────────────────────────────────────────
토큰 절감 (추정, 편차 큼)          일부 스킬은 개념 수준 (구현 보강 필요)
컨텍스트 절감 (ultra-thin-executor) 초기 설정 복잡도
재작업 감소                        학습 곡선 (2-4주)
품질 체인 자동화                   배포/모니터링 영역 공백
세션 운영 안정화                   팀 표준화 어려움

※ 수치는 내 워크플로우 기준 설계 추정. 실측 아님.
```

*발표자 노트*
"토큰 절감 수치는 실측이 아니라 설계 기반 추정치입니다. 이 점 먼저 말씀드립니다. 실제로 체감하는 효과는 재작업이 줄어든다는 것, 그리고 품질을 확인하는 체계가 생겼다는 것입니다. 한계로는 처음 설정이 복잡하다는 점, 그리고 아직 배포/운영 단계는 이 파이프라인이 완전히 커버하지 못한다는 점을 솔직하게 인정합니다."

### 8-3. 리스크와 대응

**슬라이드**
```
주요 리스크 (4개)                 우회 경로
──────────────────────────────────────────────────────────
docs/planning/tasks.md 누락   → 경로 규칙 문서화 + 수동 생성 가능
평가 코드 변조 (self-improvement-loop)  → Frozen Metric 분리 강제
품질 체인 우회                   → 오케스트레이터가 순서 강제
컨텍스트 오염                    → 세션 분리 + 핸드오프 프로토콜

이 중 상위 3개(파일 누락/변조/우회)는 단일 장애점 수준
"취약점은 있지만 우회 경로도 있다"
```

*발표자 노트*
"주요 리스크 네 가지이며, 이 중 상위 세 가지가 단일 장애점 수준입니다. docs/planning/06-tasks.md 파일이 없거나 경로가 틀리면 파이프라인이 멈추고, 평가 코드를 건드리면 self-improvement-loop 자체가 의미를 잃으며, 품질 체인을 우회하면 검증이 통째로 빠집니다. 그나마 각각 우회 경로가 있습니다. 파일은 수동으로도 만들 수 있고, 오케스트레이터는 교체 가능합니다."
