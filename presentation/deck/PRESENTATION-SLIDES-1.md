# 1장. Why?

### 1-0. 오늘 이야기하는 내용

**슬라이드**

*발표자 노트*
"오늘 이 자리는 '뭐가 되는가'가 아니라 '어떻게 쓰는가'에 대한 이야기입니다. 기초 개념이나 도입 당위성은 이미 아시는 분이 많으니 건너뜁니다. '정답 공유'가 아니라 '저렇게 쓰고 있구나' 정도로 들어주시면 됩니다."

오늘 다루는 것
  • Claude Code를 "어떻게 쓰고 설계하고 평가하는가"
  • 운영 원칙과 그 원칙을 반복 가능하게 만드는 장치(Skill)
  • 작업자 → 조율자로의 역할 변화

오늘 다루지 않는 것
  • LLM / 프롬프트 기초
  • AI 도입의 당위성 설득
  • 특정 툴의 기능 나열식 소개

### 1-1. 에이전트와 사람

**슬라이드**
```
에이전트가 막 쓸 만해진 지금, 쓰는 사람마다 결과 편차가 큰데
그 차이는 모델이 아니라 운영에서 나온다.

결국 에이전트를 잘 쓰려면
운영 원칙이 필요하고
스킬 = 운영 원칙을 반복 가능하게 고정하는 장치
```

*발표자 노트*
"모델은 누구나 같은 걸 씁니다. 그런데 결과는 사람마다 다릅니다. 그 차이가 모델이 아니라 **운영**에서 나온다는 게 오늘의 출발점입니다. 그래서 말하고 싶은 건 이 세 문장입니다.

첫째, 에이전트는 사람을 대체하지 않고 **역할 구성을 바꿉니다** — 작업자에서 조율자로. 4장에서 다룹니다.
둘째, 그냥 던지면 실패합니다. 왜 실패하는지는 2장에서 봅니다.
셋째, 스킬은 그 운영 원칙을 매번 다시 말하지 않아도 되게 **고정**하는 장치입니다.

이런 운영 원칙에 대해 Claude Code 중심 사례

### 1-2. 확률적 출력 — 베이즈

**슬라이드**
```
같은 질문에 다른 답이 나오는 이유

  모델의 출력 = 분포에서 뽑는 표본 (0/1의 확정 아님)
    ├─ 맥락 얇다 → 분포 넓다 → 답이 흔들린다
    ├─ 맥락 두껍다 → 분포 좁다 → 답이 일관된다
    └─ 운영 원칙 = 맥락을 "어떻게 **최소 비용으로** 두껍게 유지할지"

```

*발표자 노트*
"LLM 출력은 확정이 아니라 **분포에서의 표본**입니다. 0/1로 되느냐 안 되느냐의 사고를 먼저 내려놓으셔야 합니다. 맥락이 얇으면 분포가 넓어서 답이 흔들리고, 두껍게 유지하면 좁혀집니다. 오늘 이야기하는 세션 운영·스킬·핸드오프 — 전부 '맥락을 어떻게 두껍게 유지할지'입니다. 그래서 이 발표는 **정답 찾기가 아니라 분포 좁히기**에 대한 이야기입니다.맥락을 가장 쉽게 두껍게 유지하는 것은 토큰을 많이 사용하는 것입니다. 하지만 비용 효율적인 방법은 아닙니다. "

### 1-3. 앱 vs IDE vs CLI

**슬라이드**
| 구분 | Claude.ai 웹/앱 | **Claude Code (CLI)** | Cursor / Antigravity (IDE) |
|---|---|---|---|
| 로컬 파일·git·터미널 | ❌ | ✅ | ✅ |
| 세션·권한 위임 | 제한 | 세밀 제어 | 도구별 상이 |
| 확장 메커니즘 | Instructions, Projects | **Skills / MCP / Hooks / Sub-agent** | 유사 개념, 동일 메커니즘 아님 |

*발표자 노트*
"Skill·MCP·오케스트레이션은 전부 **CLI 기준**이고 앱에서는 안 됩니다. 저도 앱 → Cursor → 지금 Claude Code로 옮겨왔고, 각 도구에 유사 개념은 있지만 동일 메커니즘은 아닙니다. 이 표 하나로 끝내고 바로 넘어가겠습니다."

### 1-4. 오늘 발표의 목표

**슬라이드**
```
관점
  정답 소개 ❌ / 운영 체계 공유 ✅
  "제가 어떤 기준으로 쓰고, 어떻게 설계하고 평가하는가"

  1. 운영 감각 — 설정·흐름·주의점
  2. 평가 기준 — 내 체계가 잘 되고 있는지 보는 눈
  3. 시작점  — 어디부터, 어떤 순서로
  4. 재공유
```

*발표자 노트*
"운영 체계를 공유하고, 그걸 **냉정하게 평가하는 방법**을 같이 보는 자리입니다."
