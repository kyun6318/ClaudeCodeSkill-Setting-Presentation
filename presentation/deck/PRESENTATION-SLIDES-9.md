# LLM Tools, Skills, and AI-Native Workflows

# 9장. 스킬 도입과 개발에 있어서

### 9-1. 스킬 도입 순서

**슬라이드**
```
5개만 써도 초반 효용의 상당 부분은 확보된다

  Step 1  systematic-debugger        — 버그 분석, 위험 없음
  Step 2  pre-completion-verifier    — 완료 기준 수립
  Step 3  requirements-dialogue      — 기획 체계화
  Step 4  dependency-auto-executor   — 실행 자동화
  Step 5  self-improvement-loop      — 자율 개선

→ 도입 기준: 작게 시작, 검증부터, 점진적 확장
```

*발표자 노트*
"전부를 한 번에 도입하려 하면 실패합니다. 가장 안전한 첫 번째 스킬은 systematic-debugger입니다. 기존 워크플로우에 영향을 주지 않으면서 버그 분석의 품질을 높여줍니다. 거기서 시작해서 검증 루프를 붙이고, 기획 체계를 갖추고, 실행을 자동화하는 순서로 확장하시면 됩니다. 이 다섯 개만으로도 초반 체감은 꽤 큽니다."

### 9-2. 앞으로의 방향

**슬라이드**
```
아직 비어 있는 영역
  → 배포 파이프라인 통합
  → 운영/모니터링 단계
  → 팀 단위 표준화

진화 방향
  → 플랫폼이 흡수하는 것: 놔두기
  → 사용자 고유 워크플로우: 계속 정제
  → 새로운 실패 경험이 새 스킬로
```

*발표자 노트*
"이 체계는 계속 진화하고 있습니다. 새로운 실패를 만날 때마다 새 스킬이 생깁니다. 반대로 모델이 좋아지면서 불필요해지는 스킬은 버립니다. 고정된 시스템이 아니라 실패 경험에서 계속 업데이트되는 체계입니다."


### 9-3. 마무리 메시지

**슬라이드**
```
도입 기준은 작게 시작하되 검증부터

운영 원칙이 도구보다 먼저다

스킬은 그 원칙을 반복 가능하게 만드는 장치다

─────────────────────────────────────
오늘 공유 범위 = 슬라이드 안에 담긴 구조·포맷·축약 예시.
각자 시도해 보시고, 경험·방법을 되공유해 주세요.
```

*발표자 노트*
"처음에 드린 세 줄을 다시 드립니다. 같은 말인데 조금 다르게 들리셨으면 합니다.
에이전트가 역할을 가져간다는 건 맞습니다. 그 전환이 잘 일어나려면 무엇을 맡기고 무엇을 남길지 설계해야 합니다. 그 설계의 기준이 운영 원칙이고, 스킬은 그 원칙을 매번 처음부터 설명하지 않아도 되게 고정합니다. 이게 오늘 제가 드리고 싶었던 이야기입니다.
오늘 공유드린 건 외부 저장소가 아니라 슬라이드에 담긴 축약 예시까지입니다. 각자 손으로 한 번 써 보시고, 각자 방식을 되공유해 주시면 저도 배울 수 있습니다. 감사합니다."
