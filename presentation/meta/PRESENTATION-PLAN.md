## 발표 목적

### 핵심 메시지 (처음과 끝에 반복)

1. 에이전트는 사람을 대체하는 도구가 아니다
2. 잘 쓰려면 운영 원칙이 필요하다
3. 스킬은 그 운영 원칙을 반복 가능하게 고정하는 장치다

### 발표 관점

> 이 발표는 장점 소개가 아니라, 운영 체계를 어떤 기준으로 설계하고 평가하는지를 함께 공유한다.

### 청자가 발표 후 가져갈 것

- **인식 전환**: LLM 도구는 "질문-답변" 도구가 아니라 "운영 체계를 갖춰야 하는 협업 도구"
- **평가 기준**: 이 체계가 잘 되고 있는지, 어디가 부족한지를 어떤 잣대로 보는지
- **실무 감각**: 어떻게 설정하고, 어떤 흐름으로 사용하고, 뭘 조심해야 하는지
- **시작점**: 어디서부터 도입할 수 있는지, 어떤 순서로 확장하는지

## 발표 전략

### 무게 배분: 전반부 가볍게, 후반부에 힘을 준다

```
전반부 (1-3장, ~19분): 개념 · 문제 인식 — 빠르게 통과
  "이미 아시는 분들이 많으니 핵심만 짚겠습니다"

중반부 (4-5장, ~25분): 업무 방식 변화 · 운영 원칙 — 공감과 전환
  "도구보다 일하는 방식이 바뀌어야 합니다"

후반부 (6-9장, ~31분): 스킬 체계 · 설정 · 효과 · 시작 — 구체적 실체
  "이것이 제가 실제로 쓰는 방식입니다" ← 여기에 힘

Q&A (~15분): 운영 방식, 도구 선택, 비용, 팀 적용 등 자유 질의
```

### 6장. 현재 스킬 묶음은 어떤 문제를 해결하는가 (8분)

| 항목 | 내용 |
|------|------|
| 원본 소스 | 03-WHY (5가지 결함 + 해결), 01-CATALOG (범주 맵), 05-NATIVE-VS-CUSTOM (3범주), 02-RELATIONSHIPS (파이프라인) |
| 슬라이드 수 | 4-5장 |
| 핵심 | 결함 → 해결 매핑 테이블, 범주 맵 한 장, 파이프라인 흐름도 한 장 |

| 주의 | 65개 스킬을 나열하지 않음. **범주 맵 이미지 1장**(전체 리스트 공개 아님). 외부 저장소 링크 없음 |
| 6.3 스킬의 수명 | 아래 참조 |

**6.3 스킬의 수명 — 발표 소재** (1슬라이드, 구두 1.5-2분 이내):

큰 지도를 보여준 직후, "이 중 전부가 영원한 건 아니다"로 전환. 깊이 들어가지 않고 흡수/고유를 한 장에 대비시킨 뒤 바로 7장으로 넘긴다.

플랫폼이 흡수할 수 있는 스킬:
- 프롬프트 고정 스킬 (python-pro, typescript-pro 등): 모델 학습 데이터가 개선되면 별도 지시 없이도 최신 패턴을 적용하게 될 가능성이 높음
- 기본 검증 (pre-completion-verifier): 이미 시스템 프롬프트에 부분적으로 반영되는 추세. 다만 사용자 환경과 검증 습관에 따라 커스텀 규칙은 여전히 남을 수 있음
- 토큰 최적화 (token-compressor): 플랫폼 자체의 출력 효율이 개선되면 역할이 줄어들 수 있으나, 프로젝트별 출력 규칙은 별도로 필요할 수 있음

사용자 고유성이 강한 스킬 (모델 발전과 무관하게 남는 것):
- 프로세스 강제 (full-pipeline-orchestrator, dependency-auto-executor, autonomous-build-harness): "이 순서로 반드시 실행하라"는 사용자의 워크플로우. 범용 제품에 특정 사용자의 파이프라인을 내장할 수 없음
- 철학적 프레임 (requirements-dialogue, technical-premise-deconstructor, trinity): "이 기준으로 질문/평가하라"는 사용자의 가치 체계
- 구조적 자가개선 (self-improvement-loop, Frozen Metric): 3파일 분리 + git keep/revert 루프는 실험 설계 방법론

> **발표 톤 주의**: "불필요해진다"가 아니라 "플랫폼이 일부 흡수할 수 있지만, 환경에 따라 남을 수 있다" 수준으로. 단정적 표현은 반박 포인트가 됨

## 작성 순서

> **원칙**: 전체 서사의 중심축(1장 메시지 + 4장 전환점)을 먼저 확정한다.
> 이 두 장의 톤과 논지가 나머지 장의 기준이 되므로, 나중에 쓰면 1-2-5장까지 재조정 위험이 생긴다.

### Phase 1: 중심축 확정 — 메시지와 전환점

발표의 핵심 메시지(1장)와 전환점(4장)을 먼저 쓴다. 4장은 5장의 전제이자 가장 고난도 신규 작성 장이므로 초기에 확정해야 한다.

```
1장 (핵심 메시지 선언) → 4장 (업무 방식 변화, 발표 전환점)
```

### Phase 2: 용어 기반 — 기술 요소 정의

3장은 5장과 7장이 전제하는 용어(Tool Calling, MCP, Skill, Sub Agent)를 정의하는 장이므로, 5장·7장보다 먼저 써야 용어 불일치를 방지한다.

```
2장 (왜 실패하나) → 3장 (기술 요소 — 용어 정의)
```

### Phase 3: 운영과 스킬 — 용어 기반 위에서 작성

3장에서 확정한 용어를 전제로 운영 방식과 스킬 구체화를 쓴다.

```
5장 (내 운영 방식) → 6장 (스킬이 해결하는 문제) → 7장 (설정과 사용) → 8장 (평가와 한계) → 9장 (시작하기와 앞으로)
```

### Phase 4: 통합 및 자료 준비

```
전체 흐름 점검 → 4-5장 연결 문장 점검 → 사전 녹화 영상 편집 → 슬라이드 내 코드/설정 스니펫 정리
```
