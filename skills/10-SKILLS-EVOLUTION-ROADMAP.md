## 구조적 약점 Top 4

### 약점 1: 배포 이후 완전 공백 (Post-Launch Void)

```
현재 커버리지:

  아이디어 → 기획 → 명세 → 개발 → 테스트 → 리뷰 → ??? 
                                                      ↑
                                              여기서 끊김

빠진 부분:

  → 빌드/패키징 → 배포 → 모니터링 → 인시던트 대응 → 운영 최적화
    (약함)        (없음)   (없음)      (없음)           (없음)
```

**위험**: 코딩 단계에서 발견한 결함은 1x 비용, 프로덕션에서 발견하면 **100x 비용**.
배포~운영 스킬이 없으면 가장 비싼 실패를 방치하는 것.

---

### 약점 2: 멀티에이전트 디버깅 부재

51개 스킬이 수십 개의 에이전트를 생성하지만, **에이전트 간 상호작용을 디버깅할 도구가 없음**.

| 있는 것 | 없는 것 |
|--------|--------|
| a2a (통신 프로토콜) | 에이전트 간 메시지 추적 |
| systematic-debugger (코드 디버깅) | 에이전트 행동 디버깅 |
| quality-gate (메트릭 수집) | 에이전트 결정 경로 추적 |
| session-report (세션 리포트) | 멀티에이전트 상관 분석 |

**위험**: dependency-auto-executor가 태스크를 자동 실행할 때, 하나가 실패하면 **왜 에이전트가 그 선택을 했는지 추적이 어려움**.

## 결론

### 현재 시스템의 성격

> **"최고의 건축가가 설계한 건물인데, 건물 관리인이 없다."**

51개 스킬은 **아이디어→코드 전환**에서는 경쟁력이 있다.
하지만 **코드→프로덕션→운영**에서는 기초적인 도구조차 없다.

### 가장 시급한 3가지

1. **ci-cd-builder** — 코드가 프로덕션에 도달하는 다리
2. **deploy-orchestrator** — 프로덕션 배포 자체
3. **e2e-test-designer** — 전체 동작 보장

### 신규 스킬 추가의 예상 효과

| 현재 (51개) | Phase 1-2 후 (57개) | Phase 1-5 후 (71개) |
|------------|--------------------|--------------------|
| 라이프사이클 커버리지 45% | 60% | 80% |
| 배포~운영 스킬 0개 | 4개 | 8개 |
| 테스트 유형 2개 (단위+브라우저) | 3개 (+E2E) | 5개 (+접근성,성능) |
| 언어 전문가 3개 | 3개 | 5개 |
| AI 검증 0개 | 0개 | 1개 (hallucination) |
