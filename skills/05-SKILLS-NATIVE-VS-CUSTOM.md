# Skills: Native vs Custom — Claude가 스킬 없이 할 수 있는 것과 없는 것

## 핵심 질문

> "이 51개 스킬은 Claude가 자체적으로 제공할 수 있는 기능이 아닌가?"

**솔직한 답: 51개 중 ~7개만 "Claude가 못하는 것"을 추가한다.**
나머지 ~44개는 "할 수 있지만 매번 시키기 귀찮거나, 시키지 않으면 안 하는 것"을 자동화한다.

---

## Claude Code 네이티브 능력

Claude Code는 스킬 없이도 다음을 수행할 수 있다:

| 능력 | 네이티브 도구/방법 |
|------|------------------|
| 기획/질문 | 자연어로 "코딩 전에 질문해줘" 요청 |
| 코드 리뷰 | "이 코드 리뷰해줘" |
| 디버깅 | "이 에러 분석해줘" |
| 테스트 작성 | "테스트 만들어줘" |
| 서브에이전트 병렬 실행 | Agent 도구 네이티브 지원 |
| 태스크 관리 | TaskCreate / TaskUpdate 내장 |
| 계획 모드 | EnterPlanMode 내장 |
| 보안 검토 | "보안 취약점 확인해줘" |
| 리서치 | WebSearch / WebFetch 내장 |
| 파일 검색/편집 | Read / Edit / Grep / Glob 내장 |
| 명령어 실행 | Bash 내장 |

## 3가지 범주 분류

> **주의: 이 범주들은 배타적 분류(partition)가 아니라 중첩 가능한 태그(tag)다.**
> 하나의 스킬이 "프롬프트 고정"이면서 동시에 "프로세스 강제"일 수 있고,
> "프로세스 강제"이면서 "진짜 새 능력"일 수도 있다.
> 각 스킬의 **가장 핵심적인 가치**를 기준으로 주(primary) 범주를 배정했으며,
> 중복 해당하는 스킬은 ※로 표시한다.

### 범주 1: 프롬프트 고정 (Prompt Crystallization) — ~26개 (주 범주 기준)

**본질**: Claude에게 매번 같은 지시를 내리는 대신, 스킬로 한 번 정의하여 재사용.
**가치**: 편의성 + 품질 균일화.

#### 왜 필요한가

```
스킬 없이:
  사용자: "코딩 전에 먼저 질문해줘. 10개 명확성 차원으로 분석하고,
          각 차원의 모호함 점수를 매겨서 0.15 이하가 될 때까지 질문을 계속해.
          경쟁 분석도 하고 페르소나도 정의하고 비즈니스 모델도 검증해.
          최종적으로 9개 기획 문서를 생성해줘..."  ← 매번 500자 이상

스킬 있음:
  사용자: "/requirements-dialogue"  ← 한 마디
```

| 스킬 | 고정된 프롬프트의 핵심 |
|------|---------------------|
| requirements-dialogue | "10개 명확성 차원 × 40-50개 질문으로 기획해" |
| idea-concretizer | "재귀적으로 사고해서 3-4개 MVP 제안해" |

| the-fool | "5가지 모드(반증, 레드팀 등)로 기획을 공격해" |
| council-codex | "council + Codex MCP 정적 리뷰 루프 돌려" |
| task-decomposer | "Domain-Guarded 구조로 TASKS.md 생성해" |
| code-reviewer | "2단계로 리뷰해: 스펙 준수 → 코드 품질" |
| quality-gate | "커버리지/복잡도/lint/타입 에러 메트릭 측정해" |
| sync | "명세와 코드의 구조적 일치를 검증해" |
| systematic-debugger | "4단계(증거→가설→실험→검증)로 분석해" |

#### 이 범주의 가치

> `/requirements-dialogue` 한 마디 = 500자 프롬프트의 **정확한 재현**.
> 10번 실행하면 10번 같은 품질. 프롬프트를 기억하거나 복사할 필요 없음.

### 범주 2: 프로세스 강제 (Process Enforcement) — ~7개 (주 범주 기준)

**본질**: Claude가 자발적으로 하지 않는 행동을 **건너뛸 수 없게** 강제.
**가치**: 일관성 + 누락 방지.

#### 왜 필요한가

Claude의 자연 행동에는 구조적 편향이 있다:

```
Claude 자연 행동                    스킬이 강제하는 행동
─────────────────                   ─────────────────────
"바로 코딩 시작"               →    "6단계 순서대로만 진행"
"한 번 리뷰하고 끝"            →    "verification→quality-gate→code-reviewer 체인"
"실패하면 보고만"              →    "5사이클 자동 수정"
"범위가 자연스럽게 확장"       →    "Drift Anchor 이탈 시 경고"
"편한 것부터 실행"             →    "의존성 분석 → 최적 순서 강제"
"완료라고 선언"                →    "증거 없으면 선언 차단"
```

#### 해당 스킬 목록

| 스킬 | 강제하는 프로세스 | Claude의 자연 행동 (스킬 없이) |
|------|-----------------|-------------------------------|
| **full-pipeline-orchestrator-pipeline** ※범주3 겸 | Phase별 모델 배정(Opus/Sonnet/Gemini) + Drift Anchor 강제 | 단일 모델로 전부 처리, 범위 이탈 허용 |
| **dependency-auto-executor** | TASKS.md 의존성 그래프 순서대로 실행 + Phase별 품질 게이트 | 순서 무시하고 쉬운 것부터 |
| **autonomous-build-harness** ※범주3 겸 | Build→Eval 피드백 루프 강제 (1/3/5/custom 사이클, 사용자 선택) | 한 번 빌드하고 끝 |
| **requirements-dialogue-anchored** | Phase 0.3에서 objective/scope/constraint 불변 확정 | 범위가 대화 중 자유롭게 확장 |
| **powerqa** | 실패 시 자동 수정 5사이클 강제 (동일 실패 3회 시 중단) | 실패 보고 후 사용자 지시 대기 |
| **desktop-bridge** | GitHub Issue 기반 Desktop↔CLI 핸드오프 강제 | 수동으로 컨텍스트 전달 |

※ **quality chain**은 스킬이 아니라 `dependency-auto-executor/references/quality-chain.md`에 정의된 품질 절차다. dependency-auto-executor가 Phase 완료 시 자동 실행하는 순서를 규정한다. sync는 이 자동 체인이 아닌 수동 검증(PR 생성 전/사용자 요청 시).

#### 이 범주의 가치

> Claude는 "시키면 하지만, 시키지 않으면 건너뛴다."
> 스킬은 "건너뛸 수 없게" 만든다.
>
> **프로세스의 일관성이 품질을 만든다.**
> 10번 중 10번 같은 검증을 거쳐야 신뢰할 수 있다.

### 범주 3: 진짜 새로운 능력 (Genuine New Capabilities) — ~7개 (일부 범주 2와 중복)

**본질**: 프롬프트만으로는 구현 자체가 불가능한 구조적/시스템적 능력.
**가치**: 불가능 → 가능.

#### 해당 스킬 목록

| 스킬 | 왜 프롬프트로 불가능한가 |
|------|----------------------|
| **self-improvement-loop (Frozen Metric + 2계층 RSI)** | 평가 코드(prepare.py)를 **불변으로 분리**하고, target.py만 수정 → git keep/revert 루프를 돌리는 구조. "점수를 조작하지 못하게" 하려면 파일 분리 + git 워크플로우가 필요. "자기 평가를 속이지 마"라는 프롬프트는 작동하지 않는다 — 구조적 강제만 작동한다. |
| **autonomous-build-harness (Build-Eval 피드백 루프)** ※범주2 겸 | builder와 evaluator가 **격리된 에이전트**로 분리되어야 evaluator의 "회의적 기본값"이 builder의 자신감에 오염되지 않는다. 같은 에이전트가 만들고 평가하면 자기 코드에 관대해진다. 사이클 수는 사용자 선택(1/3/5/custom). |
| **token-compressor (session hook)** | output-optimizer.js를 Claude Code hooks로 등록하여 **세션 레벨에서** CLI 출력을 자동 압축하는 것은 설정 파일 기반 시스템이다. 프롬프트가 아니라 hooks 인프라가 필요하다. |
| **full-pipeline-orchestrator-pipeline (Phase별 모델 라우팅)** ※범주2 겸 | Phase 1은 Opus, Phase 4는 Sonnet, UI는 Gemini MCP — **모델을 Phase별로 명시적으로 전환**하는 것은 프롬프트가 아니라 실행 전략이다. |
| **cost-model-router (모델 자동 선택)** | 태스크 복잡도를 분석하여 Haiku/Sonnet/Opus를 **자동 라우팅**하고, 실패 시 상위 모델로 **자동 에스컬레이션**하는 것은 메타 실행 전략이다. |
| **a2a (에이전트 통신 프로토콜)** | Request/Response, Handoff, Broadcast 패턴의 **구조화된 에이전트 간 통신 규약**. Claude 네이티브 Agent 도구는 단방향 위임만 지원한다. |
| **kongkong2 (Query Repetition)** | 입력을 `<QUERY>task</QUERY><QUERY>task</QUERY>` 구조로 **2번 반복 주입**하는 것은 프롬프트 구조 자체의 변형이다. 사용자가 매번 수동으로 할 수 있지만, 모든 에이전트 호출에 자동 적용하는 것은 시스템 레벨 패턴이다. |

#### 이 범주의 가치

> 프롬프트로 "자기 평가를 속이지 마"라고 하면 Claude는 "네, 알겠습니다"라고 하고 속인다.
> Frozen Metric으로 평가 코드를 분리하면 **구조적으로 속일 수 없다**.
>
> "행동 지시"와 "구조적 강제"의 차이.

## 비유로 정리

| 범주 | 비유 | 설명 |
|------|------|------|
| 프롬프트 고정 | **요리 레시피북** | 셰프(Claude)는 요리할 수 있다. 레시피가 있으면 매번 같은 맛을 낸다. |
| 프로세스 강제 | **조리 매뉴얼 + 체크리스트** | 셰프가 바쁘면 손씻기를 건너뛴다. 매뉴얼이 강제하면 건너뛸 수 없다. |
| 진짜 새 능력 | **새로운 조리 도구** | 오븐이 없으면 빵을 구울 수 없다. 프라이팬으로 아무리 잘해도 안 된다. |

## 결론

> **51개 스킬은 "Claude가 못하는 것"을 만드는 게 아니라,
> "Claude가 할 수 있지만 안 하는 것"을 반드시 하게 만드는 시스템이다.**
>
> ~7개의 진짜 새 능력이 기반을 깔고 (일부는 프로세스 강제도 겸함),
> ~7개의 프로세스 강제가 품질을 보장하고,
> ~26개의 프롬프트 고정이 편의성을 제공한다.
> (범주는 중첩 가능하므로 합산 ≠ 51. 각 스킬의 주 범주 기준 분류.)
>
> 이 세 계층이 함께 작동할 때,
> "할 수 있는 AI"가 "반드시 하는 AI"로 바뀐다.
